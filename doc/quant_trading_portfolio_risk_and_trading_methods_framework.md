> 建议前置阅读：[详细功能规格](./quant_trading_detailed_function_spec.md) / [量化因子库](./quant_trading_factor_library_and_implementation_guide.md)
>

## 1. 文档目标

本文档用于把 `quant_exchange` 项目中已经分散存在的三类核心能力统一整理成一份专题文档：

- 投资组合与资金管理
- 风险控制体系
- 交易方法与策略方法论

目标不是重复已有文档，而是把这三块以“可实施、可配置、可审计”的方式集中定义，方便后续：

- 做系统实现
- 做数据库与 API 设计
- 做策略上线流程
- 做风控审批与验收

## 2. 三类能力在平台中的定位

### 2.1 投资组合

投资组合模块解决的不是“单个策略赚不赚钱”，而是：

- 多个策略怎么一起运行
- 多个市场怎么分配资金
- 多个账户怎么统一看风险
- 如何控制相关性、集中度、杠杆和容量

### 2.2 风险控制

风险控制模块解决的不是“亏了以后怎么看”，而是：

- 哪些交易不允许发出
- 哪些仓位不允许继续扩大
- 哪些异常必须立即停机、减仓或平仓
- 出现系统故障时如何保护资金

### 2.3 交易方法

交易方法模块解决的不是“能不能下单”，而是：

- 策略如何产生信号
- 信号如何转成目标仓位
- 仓位如何转成订单
- 哪些方法适合股票、期货、加密货币

## 3. 投资组合与资金管理框架

### 3.1 核心目标

投资组合管理的目标应定义为：

- 在给定风险预算下最大化组合层风险调整后收益
- 降低单一策略或单一市场对组合的主导性
- 提升资金使用效率
- 控制相关性、拥挤度、流动性和回撤

### 3.2 组合管理对象

平台应支持以下组合层级：

- 标的级组合
- 策略级组合
- 账户级组合
- 市场级组合
- 全平台主组合

### 3.3 核心功能

投资组合模块应支持：

- 资金配额管理
- 风险预算管理
- 策略权重管理
- 暴露聚合
- 相关性分析
- 再平衡
- 动态资金调拨
- 组合层风控
- 组合收益归因

### 3.4 常见投资组合方法

#### 固定权重法

说明：

- 给每个策略或标的固定权重
- 实现最简单，适合 MVP

实现方法：

- 配置每个策略的 `capital_limit`
- 按日或按周检查偏离并再平衡

适用场景：

- 小规模多策略组合
- 早期验证阶段

#### 波动率目标法

说明：

- 根据资产或策略历史波动率动态调整仓位
- 波动越高，仓位越低

实现方法：

- `target_weight_i = base_weight_i * target_vol / realized_vol_i`
- 对结果再加上下限、上限和流动性约束

适用场景：

- CTA
- 趋势跟踪
- 多市场资产组合

#### 风险平价法

说明：

- 目标不是资金等权，而是风险贡献接近均衡

实现方法：

- 用协方差矩阵估计边际风险贡献
- 通过数值优化找到接近等风险贡献的权重

适用场景：

- 中低频多资产组合
- 期货与股票混合组合

#### 均值方差法

说明：

- 在收益预期和协方差矩阵基础上做最优化

实现方法：

- 输入预期收益向量 `mu`
- 输入协方差矩阵 `Sigma`
- 求解目标：
  `max w^T mu - lambda * w^T Sigma w`

适用场景：

- 因子型股票组合
- 有稳定 alpha 预测值时

注意：

- 对输入误差很敏感
- 必须加权重约束、换手约束和行业约束

#### 分层风险平价 / HRP

说明：

- 通过聚类降低协方差矩阵不稳定性

实现方法：

- 对相关矩阵聚类
- 按树状结构分配风险权重

适用场景：

- 标的数量较多的股票组合
- 相关性结构较复杂的多资产组合

#### Kelly 上限法

说明：

- 从资金增长率角度给出理论最优杠杆

实现方法：

- 用策略胜率、盈亏比或 alpha / variance 推导仓位
- 实盘中只使用 fractional Kelly

适用场景：

- 单策略资金管理

注意：

- 必须设很强的上限
- 不适合直接裸用

### 3.5 组合约束

平台应支持以下组合约束：

- 单策略最大资金占比
- 单标的最大权重
- 单行业最大权重
- 单市场最大权重
- 单币种最大权重
- 多空净暴露限制
- 总杠杆限制
- 相关性上限
- 流动性容量约束
- 最大换手限制
- 最大回撤约束

### 3.6 再平衡方法

平台建议支持：

- 定时再平衡
- 阈值触发再平衡
- 风险触发再平衡
- 事件触发再平衡

实现方法：

- 组合层先生成目标权重
- 目标权重转成目标仓位
- 目标仓位再转成订单篮子
- 下单前再经过交易成本和风控裁剪

### 3.7 投资组合实现建议

推荐分三层实现：

1. `Portfolio Allocator`
   - 计算策略和标的目标权重
2. `Exposure Aggregator`
   - 聚合账户、市场、币种、行业暴露
3. `Rebalance Executor`
   - 把目标偏差转成可执行订单

MVP 建议：

- 固定权重
- 波动率目标缩放
- 基础相关性监控
- 基础组合暴露报表

## 4. 风险控制框架

### 4.1 风控目标

风控体系必须覆盖以下目标：

- 防止错误信号直接转成灾难性仓位
- 防止单一市场、单一策略、单一账户失控
- 防止系统故障继续放大风险
- 防止流动性不足时硬冲成交
- 防止模型漂移和数据异常引发错误交易

### 4.2 风控分层

平台应至少具备以下五层风控：

- 订单级风控
- 持仓级风控
- 账户级风控
- 组合级风控
- 系统级风控

### 4.3 订单级风控

应支持：

- 单笔下单数量上限
- 单笔下单金额上限
- 价格偏离限制
- 最小成交额校验
- 重复下单限制
- 下单频率限制
- 市场状态检查
- 时段检查
- T+1 / 可卖数量检查
- 涨跌停或价格保护检查

实现方法：

- 订单请求进入 OMS 前先过 `PreTrade Risk Engine`
- 拒单返回明确原因码
- 所有拒单写入 `risk_events`

### 4.4 持仓级风控

应支持：

- 单标的最大仓位
- 单标的最大杠杆
- 单标的最大亏损
- 方向性集中暴露限制
- 临近交割或退市强制降仓
- 卖空与融券可用性检查

实现方法：

- 实时持仓快照 + 市场价格
- 每次订单前重算预测持仓
- 若超限则裁剪或阻断

### 4.5 账户级风控

应支持：

- 日内亏损阈值
- 周度 / 月度回撤阈值
- 保证金率阈值
- 账户总杠杆上限
- 当日成交次数上限
- 当日换手率上限
- Pattern Day Trader 风险监控
- 券商或交易所异常状态检查

实现方法：

- 账户维度实时计算权益、保证金、buying power、可用余额
- 超限时触发：
  - 告警
  - 停止开新仓
  - 只允许减仓
  - 自动减仓

### 4.6 组合级风控

应支持：

- 多策略相关性阈值
- 行业集中度阈值
- 单市场暴露阈值
- 单币种暴露阈值
- 组合 VaR / Expected Shortfall
- 组合最大回撤阈值
- 组合 liquidity stress 阈值

实现方法：

- 定期与实时双轨计算
- 组合级超限时回写到各策略可用资金和可用仓位

### 4.7 系统级风控

应支持：

- 行情中断
- 订单回报中断
- 时钟漂移
- 数据延迟超阈值
- 适配器异常
- 数据字段异常
- 模型服务不可用

实现方法：

- 建立健康检查和熔断中心
- 异常时自动降级为：
  - 停止新开仓
  - 只允许平仓
  - 全局 kill switch

### 4.8 风控动作分级

建议统一风控动作级别：

- `INFO`
- `WARN`
- `BLOCK`
- `REDUCE`
- `LIQUIDATE`
- `HALT`

### 4.9 高级风控方法

平台后续应支持：

- VaR / CVaR
- 压力测试
- 情景分析
- 流动性冲击测试
- 相关性上升情景测试
- Gap risk 模拟
- 极端波动自动降杠杆
- 模型漂移触发减权

### 4.10 风控实现建议

推荐拆分为四个部件：

1. `PreTrade Risk Engine`
2. `RealTime Risk Monitor`
3. `Risk Event Handler`
4. `Kill Switch Controller`

MVP 建议先实现：

- 单笔下单限制
- 单标的仓位限制
- 单策略仓位限制
- 单日亏损限制
- 最大回撤限制
- 保证金率预警
- 行情中断停单
- 一键 kill switch

## 5. 交易方法框架

### 5.1 交易方法的分层

平台应把交易方法拆成四层：

- Alpha 层
- 组合与仓位层
- 执行层
- 风控覆盖层

也就是说，交易方法不是只有“信号怎么来”，还包括：

- 仓位怎么定
- 订单怎么下
- 风险怎么控

### 5.2 主流交易方法分类

#### 趋势跟踪

说明：

- 假设价格趋势具有延续性

常见信号：

- 均线突破
- 通道突破
- 时间序列动量
- ADX 趋势确认

适用市场：

- 期货
- 加密货币
- 股票中强趋势标的

实现方法：

- 因子层输出趋势分数
- 仓位层按波动率缩放
- 执行层分批进场或突破入场

#### 均值回归

说明：

- 假设价格偏离均值后将回归

常见信号：

- Bollinger zscore
- RSI 超买超卖
- 价差 zscore
- 日内反转

适用市场：

- 股票短周期
- 某些期货价差
- 做市周边策略

实现方法：

- 用偏离度触发入场
- 用回归到均值或止损线退出
- 必须强依赖流动性和交易成本控制

#### 横截面选股 / 选币 / 选品种

说明：

- 在同一时点对多个标的排序，买强卖弱或买优卖劣

常见信号：

- 动量排序
- 价值排序
- 质量排序
- 情绪排序
- 多因子综合评分

适用市场：

- 股票最典型
- 也适用于期货和加密货币的多标的选择

实现方法：

- 截面 rank
- 多因子加权
- 行业或品种中性化
- 再做组合构建

#### 事件驱动

说明：

- 围绕公告、财报、政策、清算、上币、退市、宏观数据等事件交易

常见信号：

- earnings surprise
- 政策语气变化
- 上市 / 退市事件
- 链上大额转账
- 爆仓集中

适用市场：

- 股票
- 加密货币
- 宏观期货

实现方法：

- 实时情报模块抽取事件
- 事件分类后输出冲击分数和时间窗口
- 策略层决定是否参与及如何衰减

#### Carry / 基差 / 期限结构

说明：

- 赚取期限结构、资金费率或展期收益

常见信号：

- basis
- annualized basis
- roll yield
- funding rate
- term structure slope

适用市场：

- 期货
- 加密货币衍生品

实现方法：

- 对期限结构标准化
- 结合流动性、保证金和挤仓风险做过滤

#### 统计套利 / 配对交易

说明：

- 利用相对价格偏离而非绝对方向交易

常见信号：

- 协整价差
- 残差 zscore
- 跨市场价差
- A/H、现货/期货、perp/spot 偏差

适用市场：

- 股票
- 期货
- 加密货币

实现方法：

- 先做配对和稳定性筛选
- 实时计算 spread 和 zscore
- 结合成交约束和相关性失效保护

#### 做市与流动性提供

说明：

- 通过双边报价赚取价差和返佣

常见信号：

- mid price
- spread
- inventory imbalance
- order book imbalance
- volatility regime

适用市场：

- 加密货币
- 期货
- 部分股票通道

实现方法：

- 报价引擎
- inventory 风控
- gamma-like inventory control
- 自动撤单和刷新机制

#### Breakout + Confirmation

说明：

- 突破本身不够，要和量能、波动、OI、情绪确认结合

实现方法：

- `breakout_pct + volume_ratio + sentiment_score + oi_change`
- 做成多条件入场逻辑

适用市场：

- 股票
- 期货
- 加密货币

### 5.3 仓位管理方法

平台应支持：

- 固定手数
- 固定金额
- 固定风险金额
- 波动率目标仓位

- 分批建仓

### 5.4 退出方法

平台应支持：

- 固定止损

- 移动止损
- 时间止损
- 波动率止损
- 信号反转退出
- 组合层统一减仓
- 风控强制平仓

### 5.5 执行方法

平台应支持：

- 市价单
- 限价单
- Post-only
- Reduce-only
- TWAP
- VWAP
- POV
- Iceberg
- 分批挂单
- 智能路由

### 5.6 不同市场推荐的交易方法组合

#### A 股

- 横截面多因子选股
- 事件驱动
- 中低频趋势
- 集合竞价增强

#### 港股

- 价值 + 质量
- 南向资金因子
- A/H 溢价或 ADR 映射
- 事件驱动

#### 美股

- 多因子选股
- 财报事件驱动
- 盘前盘后缺口交易
- 风格轮动

#### 期货

- 趋势跟踪
- Carry
- 期限结构
- 跨期套利
- 宏观 regime filter

#### 加密货币

- 趋势跟踪
- Funding / basis carry
- 链上与情绪驱动
- perp/spot 偏差
- 爆仓与拥挤度驱动

## 6. 系统实现建议

推荐把这三块能力接入统一流水线：

1. 因子层
   - 生成 alpha、risk、execution 因子
2. 方法层
   - 生成信号和目标仓位
3. 组合层
   - 分配资金和裁剪暴露
4. 风控层
   - 检查是否允许交易
5. 执行层
   - 把目标仓位转成订单
6. 监控层
   - 跟踪偏差、回撤、风控事件和收益归因

## 7. MVP 与增强版建议

### 7.1 MVP 必须有

- 单策略模板

- 市价 / 限价下单
- 基础止损和 kill switch

### 7.2 第二阶段应补

- 波动率目标仓位

### 7.3 增强版平台应补

- 风险平价 / HRP
- 多市场统一组合引擎
- 高级执行算法

## 9. 结论

这三类能力都不是附属模块，而是量化平台的核心中枢：

- 交易方法决定有没有 alpha
- 投资组合决定 alpha 能不能稳定转成收益
- 风险控制决定收益能不能活下来

因此在实现顺序上，推荐始终遵循：

1. 先定义交易方法和仓位逻辑
2. 再定义组合分配和暴露约束
3. 最后用平台级风控把所有链路兜住

只有这三者一起成立，量化平台才真正具备长期可运行性。
