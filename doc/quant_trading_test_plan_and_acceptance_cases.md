## 1. 文档目的

本文档用于把主需求文档中的需求编号体系，转化为可执行的测试计划与验收矩阵。

本文档解决以下问题：

- 每个需求编号由哪些测试项验证
- 哪些测试可以自动化，哪些必须通过集成测试或 `UAT` 验收
- 当前代码仓库已覆盖哪些需求，哪些仍处于规划中
- 如何从“需求完成”推进到“测试通过”

## 2. 编号与口径

### 2.1 需求编号来源

本文档与 [quant_trading_requirements_analysis.md](./quant_trading_requirements_analysis.md) 完全对齐，所有需求编号以主需求文档为唯一来源。

### 2.2 测试编号规则

- `TC-MD-01`：市场数据模块测试项
- `TC-ST-01`：策略研究模块测试项
- `TC-SW-01`：股票工作台测试项
- `TC-FT-01`：期货交易工作台测试项
- `TC-UI-01`：前端 UI / 主题增强测试项
- 其他模块同理

说明：

- 一个测试项可以覆盖一个或多个需求编号
- 每个需求编号至少需要出现在一条测试映射记录中

### 2.3 状态定义

- `自动化已覆盖`：仓库已有自动化测试覆盖核心行为
- `部分自动化`：已有自动化基础，但还缺异常路径 / 性能 / UAT 场景
- `计划中`：当前未实现或未编写测试，已纳入后续补齐计划
- `人工/UAT`：需要联调、产品验收或人工评审配合

## 3. 测试分层

### 3.1 单元测试

适用于：

- 计算函数
- 因子函数
- 风控规则判断
- 状态机迁移
- 数据转换与标准化函数

### 3.2 集成测试

适用于：

- 数据接入与落库
- 控制面接口
- 订单状态流转
- 模拟盘和回测链路
- 页面与后端接口联动

### 3.3 端到端测试

适用于：

- 选股工作台完整操作流
- 学习中心闭环
- 模拟交易闭环
- 下载中心断点续传闭环
- 多用户隔离闭环
- 期货交易工作台操作流（合约浏览 → 详情 → K 线 → 周期切换）
- 加密货币研究页操作流
- 暗色主题下各标签页视觉一致性

### 3.4 稳定性与故障注入

适用于：

- 断线恢复
- 下载续传
- 状态恢复
- 幂等与补偿
- 告警与风控联动

### 3.5 验收测试

适用于：

- `MVP` 上线前验收
- 增强模块进入正式排期前的 `Go / No-Go` 检查
- 专项页面体验验收
- 暗色交易终端主题视觉验收
- 期货交易页面功能完整性验收
- 三大市场（股票、加密货币、期货）统一体验验收

## 4. 测试环境

| 环境 | 目标 | 说明 |
| --- | --- | --- |
| `local` | 单元与本地联调 | 快速回归与函数级验证 |
| `test` | 模块集成测试 | 使用测试数据和仿真依赖 |
| `uat` | 页面与控制面联调 | 验证真实用户链路和多角色流程 |
| `paper` | 模拟盘长跑 | 验证接近实盘的订单 / 风控链路 |
| `prod-like` | 灰度前检查 | 接近生产的部署、监控和恢复机制 |

`MVP` 上线前必须满足以下要求：

1. 所有 `P0` 需求至少完成一条通过的验证记录
2. 所有 `P0` 高风险链路必须包含异常用例
3. 模拟交易、风控、权限和多用户隔离必须存在自动化测试
4. 网页核心链路必须完成 `E2E/UAT`
