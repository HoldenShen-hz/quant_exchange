## 1. 文档目标

本文档用于补充说明 `quant_exchange` 平台如何正式支持股票量化交易，重点覆盖：

- A 股
- 港股
- 美股

目标不是只在需求里增加一句“支持股票”，而是把股票市场对平台设计真正有影响的约束讲清楚，包括：

- 市场接入方式
- 交易时间与交易日历
- 下单单位与交易规则
- 公司行为处理
- 结算与风控口径
- 程序化交易合规与审计要求

## 2. 支持范围定义

### 2.1 平台目标支持的股票市场

| 市场 | 是否纳入平台目标 | 说明 |
| --- | --- | --- |
| A 股 | 是 | 重点关注上交所 / 深交所现金股票场景 |
| 港股 | 是 | 重点关注联交所现金股票场景 |
| 美股 | 是 | 重点关注 NYSE / Nasdaq 现金股票场景 |

### 2.2 第一阶段建议范围

为了控制交付风险，股票市场的第一阶段建议聚焦以下范围：

- 现金股票
- 日线 / 分钟线驱动策略
- 市价 / 限价等基础订单类型
- 单券商或单执行通道
- 公司行为、交易日历、停复牌基础支持

第一阶段建议暂不强行纳入：

- 期权
- OTC / 粉单
- 复杂融券借券管理
- 跨市场股票套利
- 高频逐笔撮合级仿真

## 3. 三类股票市场的关键差异

### 3.1 交易时间与交易日历

股票市场不是统一的 7 x 24 市场，A 股、港股、美股的交易时间和盘中阶段明显不同。

#### A 股

根据上交所交易规则和投资者教育材料，A 股存在开盘集合竞价、连续竞价和收盘集合竞价等分阶段交易时段，且交易日严格跟随内地市场日历。

这意味着平台必须支持：

- 盘前集合竞价
- 盘中连续竞价
- 收盘集合竞价
- 节假日与休市管理
- 午间休市

#### 港股

根据 HKEX 官方交易时间说明，港股包含开市前时段、连续交易时段、午休、以及收市竞价时段。

这意味着平台必须支持：

- Pre-opening session
- Continuous trading session
- Extended morning session / 午间结构
- Closing auction session
- 半日市与特殊节假日处理

#### 美股

根据 NYSE 与 Nasdaq 官方时间说明，美股除常规交易时段外，还普遍存在盘前和盘后时段，但是否可交易、可交易品种以及订单限制通常还取决于券商通道。

这意味着平台必须支持：

- 常规交易时段
- 盘前 / 盘后扩展时段
- 美东时区与夏令时切换
- 券商级扩展时段可交易能力配置

### 3.2 下单单位与最小交易单位

#### A 股

根据上交所交易规则，A 股竞价买入通常以 `100` 股为一手整数倍申报，零股卖出也有单独处理要求。

平台要求：

- `lot_size` 不能默认写死为 `1`
- 下单前必须做最小申报数量校验
- 零股卖出必须按交易所或券商规则处理

#### 港股

根据 HKEX FAQ，港股的 board lot 不由交易所统一固定，而是由发行人设定，不同股票的 board lot 可能不同，碎股处理也需要单独考虑。

平台要求：

- 标的主数据必须记录 board lot
- 订单校验必须以单标的 lot 规则为准
- 碎股和整手订单不能混为一谈

#### 美股

美股的最小交易单位通常由券商和通道能力决定，平台不应假设所有通道都只允许“整手”交易，也不应把港股 board lot 逻辑直接复用到美股。

平台要求：

- 最小数量和价格步进必须以 broker metadata 为准
- 扩展时段下单限制要按券商能力配置

### 3.3 结算周期与回转交易限制

#### A 股

根据上交所交易规则，竞价交易买入的股票在交收完成前不得卖出，这意味着平台在普通现金股票场景下必须支持典型的 `T+1` 约束。

平台要求：

- 可卖数量必须与持仓数量区分
- 风控和下单前检查必须识别 `T+1` 可卖限制
- 模拟盘和回测不能默认允许无限制日内回转

#### 港股

港股现金股票一般允许日内回转交易，但账户层结算、交收费和资金可用性口径需要按券商和 CCASS 规则建模。

平台要求：

- 允许“持仓可卖”和“资金可用”分开计算
- 结算周期不能硬编码为单一规则

#### 美股

美股现金股票通常允许日内买卖，但保证金账户中的日内交易频率和资金要求可能触发 FINRA 的 Pattern Day Trader 规则。

平台要求：

- 账户模型必须区分现金账户和保证金账户
- 风控规则必须支持按账户类型限制日内回转频率
- 不同券商对预借、融券、margin buying power 的定义要适配

### 3.4 涨跌停、波动控制与停复牌

#### A 股

A 股需要兼容涨跌幅限制、临时停牌、特殊板块差异、盘后固定价格或特殊撮合阶段等规则。

平台要求：

- 标的维度记录 price limit profile
- 下单引擎识别不可成交价格区间
- 风控模块识别涨跌停导致的流动性失真

#### 港股

港股需要兼容 Closing Auction Session、Volatility Control Mechanism 以及半日市等机制。

平台要求：

- 执行层必须识别不同交易阶段的有效订单行为
- 回测层需要对收市竞价和波动控制做可配置简化建模

#### 美股

美股需要兼容盘前盘后流动性下降、交易暂停、熔断以及不同 venue 的可执行性差异。

平台要求：

- 市场状态服务必须能识别 halt / resume
- 执行路由必须识别常规时段和扩展时段

### 3.5 公司行为

股票市场与加密货币、期货最大的底层差异之一，是公司行为不可忽略。

平台必须支持至少以下公司行为：

- 现金分红
- 送股 / 配股
- 拆股 / 合股
- 股票更名
- 代码变更
- 停牌 / 复牌
- 退市

平台要求：

- 研究数据必须明确复权口径
- 回测数据必须支持前复权、后复权或不复权
- 实盘持仓、成本和报表必须按真实公司行为调整
- 公司行为事件必须可追溯、可回放

### 3.6 程序化交易合规与审计

#### A 股

A 股程序化交易存在明确的报告、监测和合规要求，平台在设计上必须预留：

- 程序化交易标识
- 策略与账户关联记录
- 报告信息留痕
- 高频或异常交易监测指标
- 审批和人工复核链路

#### 港股与美股

港股和美股虽然规则体系不同，但同样不能把量化系统当作“只会发单的脚本”。平台必须保留：

- 订单决策来源
- 参数变更记录
- 异常交易告警
- 人工接管和 kill switch 记录

## 4. 对平台模块的影响

### 4.1 市场接入层

股票市场接入不能只按“交易所 WebSocket + REST”来想。很多实际场景是：

- 券商 API
- 经纪商 OMS / DMA 网关
- 市场数据供应商 + 券商执行通道分离

因此平台要求：

- 市场数据 adapter 与执行 adapter 解耦
- 允许同一市场的数据与交易来自不同通道
- 允许按券商配置下单能力、盘前盘后能力、卖空能力

### 4.2 参考数据层

必须维护：

- 证券基础信息
- 上市市场与市场区域
- 计价货币
- board lot / 最小交易单位
- 价格步进
- 交易时段
- 结算周期
- 公司行为
- 卖空 / 融资能力标签

### 4.3 回测与研究层

股票量化如果没有公司行为和停复牌处理，回测会严重失真。

必须支持：

- 交易日历驱动回测
- 公司行为调整
- 停复牌处理
- 多币种收益换算
- 复权与不复权切换
- 市场阶段限制下的信号执行模拟

### 4.4 执行层

必须支持：

- 标的级 lot 校验
- 市场时段校验
- 盘前盘后订单可执行性判断
- T+1 可卖校验
- 涨跌停或价格保护校验
- 券商错误码与内部错误码映射

### 4.5 风控层

必须支持：

- 单票集中度限制
- 行业或主题暴露限制
- 市场停牌风险识别
- 公司行为窗口风险识别
- 保证金账户日内交易频率监控
- 融券可用性和回补风险识别

### 4.6 报表与复盘层

必须支持：

- 交易收益和公司行为收益拆分
- 股息收益单独归因
- 多市场多币种净值换算
- 停牌、涨跌停、流动性不足导致的未成交归因
- 实盘与回测偏差分析

## 5. 数据模型补充要求

为了让平台真正支持 A 股、港股、美股，核心数据模型至少需要支持以下概念：

| 类别 | 关键字段或概念 | 用途 |
| --- | --- | --- |
| 市场归属 | `market_type` / `market_region` | 区分股票市场与地域规则 |
| 标的类型 | `instrument_type=EQUITY` | 区分股票与现货、期货、永续 |
| 交易单位 | `lot_size` / `min_qty` | 支持 A 股、港股、美股不同最小交易规则 |
| 结算规则 | `settlement_cycle` | 支持 `T+1` / `T+2` 等口径 |
| 交易规则 | `trading_rule_profile` | 记录盘前盘后、涨跌停、竞价阶段、board lot 等 |
| 公司行为 | `corporate_action_profile` | 记录分红、拆股、停牌、退市等 |
| 卖空能力 | `short_sellable` | 区分可融券 / 不可融券 |
| 交易日历 | `trading_calendar` | 支持节假日、半日市、夏令时 |

## 6. 推荐实施顺序

### 阶段 1：底层兼容

- 统一 `EQUITY` 标的模型
- 增加交易日历和公司行为模型
- 统一券商 / 经纪商 adapter 抽象

### 阶段 2：单市场闭环

从 A 股、港股、美股中选择一个先跑通：

- 参考数据
- 行情采集
- 回测
- 模拟盘
- 实盘执行
- 风控与报表

### 阶段 3：多市场统一

- 增加第二、第三个股票市场
- 统一多市场持仓视图
- 统一多币种净值与报表
- 统一公司行为处理和复盘分析

## 9. 结论

要真正支持 A 股、港股、美股，平台必须从“多市场多规则引擎”的角度来设计，而不是把股票当成“没有到期日的现货”。

股票量化的本质难点不在于行情本身，而在于以下几点：

- 交易时间和交易阶段不同
- 结算规则不同
- 公司行为不可忽略
- 券商通道与市场数据通道可能分离
- 程序化交易有合规和审计要求

因此，股票支持一旦纳入正式范围，后续所有实现级文档都应同步兼容：

- 股票标的模型
- 股票市场交易日历
- 公司行为数据
- 券商执行通道
- 多市场风控与报表
