## 1. 文档目标

本文档用于补充说明 `quant_exchange` 平台在市场范围上如何正式支持：

- 加密货币量化交易
- 标准化期货量化交易

目标不是简单地把“支持期货”和“支持加密货币”写成一句口号，而是明确这两类市场在数据、策略、执行、风控、报表和架构上的关键差异，并给出平台层面的兼容设计要求。

## 2. 支持范围定义

### 2.1 平台目标支持的市场类型

| 市场类型 | 是否纳入平台目标 | 说明 |
| --- | --- | --- |
| 加密货币现货 | 是 | 典型场景是币币或法币计价现货交易 |
| 加密货币永续合约 | 是 | 需要支持资金费率、标记价格、强平风险 |
| 加密货币交割合约 | 是 | 需要支持到期和交割逻辑 |
| 标准化期货 | 是 | 需要支持交易日历、日盘夜盘、结算和展期 |
| 期权 | 后续扩展 | 不纳入本次补充重点 |
| 股票 | 是 | A 股 / 港股 / 美股详见独立股票市场支持文档 |
| DEX / 链上现货与衍生品 | 后续扩展 | 后续可增加 |

### 2.2 推荐落地顺序

平台范围现已升级为多市场，但本文件聚焦“加密货币 + 期货”这两类市场的专项设计。实施顺序仍建议分阶段：

1. 先跑通一个加密货币市场闭环或一个期货市场闭环
2. 再补齐另一类市场的接入、回测和风控差异
3. 最后实现统一组合视角、多市场调度和跨市场风险聚合

这样做的原因是：

- 多市场的共性很多，但关键差异也很多
- 如果一开始要求所有市场同时实装，MVP 风险会明显升高
- 先做兼容设计，再分批实现，是更稳妥的工程路径

## 3. 加密货币与期货市场的关键差异

### 3.1 交易时间

- 加密货币市场通常是 7 x 24 连续交易
- 标准化期货通常有交易日概念、日盘夜盘划分、节假日安排和结算窗口

平台要求：

- 统一支持连续市场和分时段市场
- 所有时间序列必须区分自然时间、交易时间和交易日
- 回测引擎必须支持按交易日历驱动，而不只是按自然时间驱动

### 3.2 合约定义

- 加密货币现货主要关注交易对、最小下单精度、手续费
- 加密货币永续关注资金费率、标记价格、保证金模式
- 加密货币交割合约和标准化期货都关注到期时间、交割和合约乘数
- 标准化期货额外关注主力合约、次主力、连续合约映射

平台要求：

- `instrument` 必须支持 `SPOT`、`PERPETUAL`、`DELIVERY_FUTURE`、`FUTURE`
- 必须记录结算资产、合约面值、合约乘数、到期时间
- 必须支持主力合约和连续合约映射

### 3.3 定价与结算

- 加密货币衍生品常使用标记价格、指数价格和资金费率
- 标准化期货通常使用结算价、盯市结算、保证金调整

平台要求：

- 报表和风控必须区分成交价、最新价、标记价、结算价
- 账户快照和权益计算必须兼容资金费率与逐日盯市
- 对账模块必须支持结算后权益变化核对

### 3.4 保证金与杠杆

- 加密货币平台常见逐仓 / 全仓模式
- 标准化期货更强调交易所或经纪商定义的保证金比例、强平和风控约束

平台要求：

- 账户模型必须支持不同保证金模式
- 风控模块必须支持不同市场的保证金率规则
- 回测和模拟盘必须能近似模拟保证金占用和风险阈值

### 3.5 到期、展期与交割

- 永续合约没有到期，但有资金费率
- 交割合约和标准化期货存在到期、交割、展期和主力切换

平台要求：

- 支持到期前预警
- 支持强制平仓、移仓和展期策略
- 支持连续合约回测与真实合约实盘分离

### 3.6 市场规则和风控限制

- 加密货币交易所更关注限频、杠杆档位、价格保护、强平
- 期货市场更关注涨跌停、交易时段、结算窗口、品种监管和持仓限制

平台要求：

- 风控规则必须按市场类型、品种类型、账户类型分层配置
- 执行模块必须识别涨跌停、不可交易时段、只减仓模式等限制

## 4. 对平台各模块的影响

### 4.1 市场数据层

必须新增或强化以下能力：

- 支持加密货币连续行情与期货交易日历行情
- 支持资金费率、标记价格、指数价格、结算价
- 支持主力合约与连续合约映射
- 支持交易日切换和夜盘处理

### 4.2 参考数据层

必须维护：

- 标的基础信息
- 合约规格
- 到期和交割信息
- 交易时段与交易日历
- 保证金相关配置
- 价格限制和最小变动单位

### 4.3 策略与研究层

必须支持：

- 加密货币 7 x 24 数据回测
- 期货按交易日历回测
- 主力连续合约研究
- 展期收益和持仓切换研究
- 资金费率因子与期限结构因子

### 4.4 执行层

必须支持：

- 不同市场的下单约束和错误码映射
- 不同市场的订单状态差异
- 到期前禁开仓或减仓策略
- 交易时段检查
- 不可撤单或撤单窗口限制的处理

### 4.5 风控层

必须支持：

- 加密货币强平风险监控
- 期货结算和保证金补足监控
- 品种持仓限制
- 交割临近风险
- 市场停盘或涨跌停限制

### 4.6 组合与报表层

必须支持：

- 按市场、按账户、按品种的收益归因
- 资金费率收益与交易收益拆分
- 展期收益与现货收益拆分
- 已实现盈亏、未实现盈亏、结算盈亏拆分
- 连续合约研究表现与真实合约执行表现分开展示

## 5. 数据模型补充要求

为了让平台真正兼容期货与加密货币，核心数据模型至少需要支持以下字段或概念：

| 类别 | 关键字段或概念 | 用途 |
| --- | --- | --- |
| 标的定义 | `instrument_type` | 区分现货、永续、交割合约、标准化期货 |
| 合约规格 | `contract_size` / `contract_multiplier` | 正确计算名义价值和盈亏 |
| 到期信息 | `expiry_at` | 处理到期、交割、禁开仓和展期 |
| 结算信息 | `settlement_asset` / `settlement_price` | 处理结算与权益变化 |
| 市场时间 | `trading_session` / `trading_calendar` | 兼容 7 x 24 和分时段市场 |
| 连续映射 | `continuous_contract_mapping` | 支持主连研究与真实合约交易 |
| 衍生品定价 | `mark_price` / `index_price` / `funding_rate` | 支持衍生品估值和风险控制 |

## 6. 回测与实盘的一致性要求

如果平台要同时支持期货和加密货币，必须特别关注“研究数据”和“实盘对象”的一致性问题。

### 6.1 对加密货币市场

- 回测中需要考虑手续费、滑点、资金费率
- 衍生品策略需要考虑标记价格和强平风险

### 6.2 对期货市场

- 回测中需要考虑交易时段、跳空、涨跌停、展期和结算
- 研究可使用连续合约，但实盘必须映射到具体合约

### 6.3 平台要求

- 回测层必须区分研究标的和执行标的
- 报表层必须解释回测收益和真实执行收益的偏差来源
- 风控层必须在真实合约维度工作，不能只基于主连或连续合约

## 7. 推荐的实现策略

### 7.1 第一阶段

- 统一抽象市场接入层
- 统一抽象标的和合约模型
- 统一抽象账户、持仓、PnL 和风控事件模型

### 7.2 第二阶段

- 先完成一个加密货币场景闭环或一个期货场景闭环
- 输出对应市场的交易所 / 柜台 adapter
- 跑通回测、模拟盘、实盘一致性校验

### 7.3 第三阶段

- 接入第二类市场
- 做统一组合视角
- 做跨市场风险归因
- 做统一研究与执行编排

## 8. 结论

“支持期货”和“支持加密货币”不能只理解成多接几个接口，而是要求平台在底层模型上兼容两类市场的根本差异。

因此，项目后续所有实现级文档和代码实现都应遵循以下原则：

- 市场接入层统一抽象
- 标的模型兼容现货与合约
- 风控模型兼容资金费率、结算、交割和展期
- 回测模型兼容连续交易市场与交易日历市场
- 报表模型能解释不同市场的收益来源和风险来源

只有这样，平台才算真正具备“期货 + 加密货币量化”支持能力，而不是表面上的多市场口号。
