## 1. 文档目标

本文档用于统一 `quant_exchange` 项目中的专业术语口径，解决以下问题：

- 不同角色对同一术语理解不一致。
- 同一个英文缩写在不同市场或不同系统模块中含义不同。
- 策略、执行、风控、数据、测试文档之间口径不统一。
- 股票、期货、加密货币三类市场的术语体系存在交叉和差异。

本文档不是简单词汇表，而是统一采用以下说明框架：

- 中文名称
- 英文名称
- 常见缩写
- 详细定义
- 易混概念或边界说明
- 在本项目中的作用

## 2. 使用方式

### 2.1 使用原则

- 本文档以本项目的系统设计和实施语境为准，不强行覆盖所有机构或交易所的历史习惯。
- 如果外部接口、交易所规则或券商文档与本文档表述不同，以外部规则为准，并在适配器层做映射。
- 缩写第一次出现在需求、架构、数据库、API、测试文档时，建议同时保留英文全称。
- 后续新增模块时，应优先补充本术语表，再补充功能文档，以避免系统口径漂移。

### 2.2 适用范围

本文档覆盖以下领域：

- 量化研究与因子建模
- 市场数据与微观结构
- 策略开发、回测、模拟盘与实盘
- 订单执行与智能路由
- 组合管理、风险控制与绩效归因
- 股票、期货、加密货币、期权与 DEX
- 市场情报、情绪分析与另类数据
- 系统架构、数据库、API、安全、测试与发布

## 4. 核心量化与研究术语

### 4.1 量化交易 / Quantitative Trading

- 中文：量化交易
- 英文：Quantitative Trading
- 缩写：Quant Trading
- 详细定义：
  量化交易是指使用规则、统计模型、机器学习模型、优化算法和自动化程序，对市场进行分析、信号生成、仓位控制、订单执行和风险管理的一整套交易方法。它的重点不是“自动下单”本身，而是让交易过程可定义、可测试、可复现、可审计。
- 易混概念：
  量化交易不等于高频交易，也不等于 AI 交易。高频只是量化交易的一种时延层级，AI 只是其中一类建模方法。
- 在本项目中的作用：
  本项目的目标不是写单个策略脚本，而是建设一个覆盖研究、回测、模拟盘、实盘、风控、监控和复盘的完整量化交易平台。

### 4.2 Alpha / 阿尔法收益

- 中文：阿尔法收益、超额收益来源
- 英文：Alpha
- 缩写：无固定缩写
- 详细定义：
  Alpha 一般表示相对于市场基准或风险暴露之后的超额收益来源。在量化场景中，Alpha 更常被理解为“可以重复利用的统计优势”，可能来自因子、模型、事件、市场微观结构、另类数据或执行优势。
- 易混概念：
  Alpha 不是一次偶然盈利，也不是回测中单次高收益。只有可持续、可解释、可复现的优势，才更接近工程意义上的 Alpha。
- 在本项目中的作用：
  因子库、情绪分析、事件驱动、订单簿因子、跨市场价差等模块，都是 Alpha 生产与管理的来源。

### 4.3 Beta / 贝塔暴露

- 中文：贝塔、系统性暴露
- 英文：Beta
- 缩写：无固定缩写
- 详细定义：
  Beta 用于衡量资产或组合相对市场整体波动的敏感度。它描述的是“跟着市场一起动”的部分，而不是策略自己创造的超额收益。
- 易混概念：
  Alpha 更关注超额收益，Beta 更关注市场暴露。高收益组合不一定有高 Alpha，可能只是承担了更高 Beta。
- 在本项目中的作用：
  组合分析、风险归因、对冲策略和市场中性策略，需要区分组合收益中哪部分来自 Alpha，哪部分来自 Beta。

### 4.4 Benchmark / 基准

- 中文：基准、比较基准
- 英文：Benchmark
- 缩写：无固定缩写
- 详细定义：
  Benchmark 是评价策略收益、风险和稳定性的参考对象。它可以是指数、ETF、无风险利率、某个行业指数，也可以是自定义组合。
- 易混概念：
  Benchmark 不是“默认市场涨跌”。不同策略必须有不同基准，例如股票选股策略常用股票指数，市场中性策略可能用现金基准或低风险基准。
- 在本项目中的作用：
  报表与复盘模块需要统一记录策略的基准配置，用于超额收益、跟踪误差和信息比率计算。

### 4.5 Signal / 交易信号

- 中文：交易信号
- 英文：Signal
- 缩写：无固定缩写
- 详细定义：
  Signal 是策略根据市场数据、因子或模型推断得出的动作建议，例如做多、做空、减仓、平仓或观望。信号通常代表方向和强度，但不一定直接等同于订单。
- 易混概念：
  信号不等于订单。信号往往先进入组合构建、仓位管理和风控模块，最后才会被翻译为订单意图。
- 在本项目中的作用：
  策略运行时的核心输出之一就是信号，后续会被转为目标仓位、订单计划和执行指令。

### 4.6 Factor / 因子

- 中文：因子
- 英文：Factor
- 缩写：无固定缩写
- 详细定义：
  因子是用于解释、预测或排序资产收益与风险的一类特征变量。它可以是技术指标、横截面排序特征、微观结构指标、资金费率、链上数据、新闻情绪或基本面指标。
- 易混概念：
  因子是更偏金融研究和经济含义的概念，而 Feature 更偏工程实现和模型输入表达。
- 在本项目中的作用：
  因子库文档、Feature Store、回测引擎、训练流水线和实时信号服务都会围绕因子进行统一计算和管理。

### 4.7 Feature / 特征

- 中文：特征
- 英文：Feature
- 缩写：无固定缩写
- 详细定义：
  Feature 是机器学习或工程系统中作为输入使用的结构化变量。它强调可计算、可复用、可落库、可服务化，而不仅仅是研究层面的经济解释。
- 易混概念：
  不是所有 Feature 都是高质量因子，有些 Feature 只是辅助字段、过滤条件或模型上下文变量。
- 在本项目中的作用：
  Feature Store 负责特征定义、版本、血缘、离线计算、在线服务和训练线上一致性管理。

### 4.8 Label / Target / 标签

- 中文：标签、目标变量
- 英文：Label、Target
- 缩写：无固定缩写
- 详细定义：
  Label 是监督学习任务中希望模型预测的目标，例如未来 5 分钟收益方向、未来 1 天超额收益、未来 30 分钟波动率分位数等。
- 易混概念：
  Label 不是未来真实价格本身，而是根据建模目标定义出来的监督信号。标签定义不合理，会直接导致模型训练方向错误。
- 在本项目中的作用：
  机器学习策略、方向预测模型、情绪方向模型和风险预警模型，都需要明确标签生成逻辑和时间对齐规则。

### 4.9 Universe / 标的池

- 中文：标的池
- 英文：Universe
- 缩写：无固定缩写
- 详细定义：
  Universe 指在某一策略、研究或模型运行时允许被观察和交易的资产集合。它不仅是名单，还包含筛选规则、更新时间、版本、地域和市场边界。
- 易混概念：
  Universe 不等于交易所全部标的，也不等于最后真正成交的标的。它是策略“可选范围”。
- 在本项目中的作用：
  股票、期货、加密货币都需要维护动态 Universe，用于统一选股、选合约、选币逻辑。

### 4.10 Screener / 筛选器

- 中文：筛选器、选标器
- 英文：Screener
- 缩写：无固定缩写
- 详细定义：
  Screener 是一套根据流动性、成交额、波动率、市值、情绪、资金费率、财务指标等条件筛选资产的规则系统。它常用于生成或更新 Universe。
- 易混概念：
  Screener 更像“规则式筛选流程”，Universe 更像“筛选完成后的可交易标的集合”。
- 在本项目中的作用：
  增强版平台中，Screener 会成为 Universe 管理模块的重要组成部分。

### 4.11 Pairlist / 交易对列表

- 中文：交易对列表
- 英文：Pairlist
- 缩写：无固定缩写
- 详细定义：
  Pairlist 是数字资产场景中常见的术语，表示策略允许交易的现货币对或合约列表，通常带有黑白名单、流动性门槛和周期更新机制。
- 易混概念：
  Pairlist 可以看作数字资产市场里的轻量版 Universe，但 Universe 的抽象更通用。
- 在本项目中的作用：
  在加密货币模块中，Pairlist 是策略选币、执行白名单和风控边界的重要输入。

### 4.12 Research Lab / 研究工作台

- 中文：研究工作台
- 英文：Research Lab
- 缩写：无固定缩写
- 详细定义：
  Research Lab 是支持 Notebook、数据访问、因子研究、特征分析、实验对比和报告输出的研究环境。它让研究员能够在标准化环境中快速完成想法验证。
- 易混概念：
  Research Lab 不是生产系统，而是研究与验证环境，但它应该与生产数据口径和工程规范尽量保持一致。
- 在本项目中的作用：
  这是从想法到回测再到上线的重要入口，关系到研究效率和研究结果可复现性。

### 4.13 Experiment Tracking / 实验追踪

- 中文：实验追踪
- 英文：Experiment Tracking
- 缩写：无固定缩写
- 详细定义：
  实验追踪指对代码版本、参数、数据切片、模型版本、指标结果、图表和输出物进行统一记录，使研究结论可以被回看、复验和比较。
- 易混概念：
  仅保存一份 Notebook 不能算实验追踪。实验追踪强调元数据管理、版本关联和结果审计。
- 在本项目中的作用：
  用于管理回测实验、训练实验、超参数实验和模型优选过程。

### 4.14 Model Registry / 模型注册表

- 中文：模型注册表
- 英文：Model Registry
- 缩写：无固定缩写
- 详细定义：
  Model Registry 用于统一管理模型定义、模型版本、依赖特征、训练数据范围、上线状态、审批状态、部署记录与回滚记录。
- 易混概念：
  训练出的模型文件不等于模型注册。注册表强调治理、状态、可审计和部署关联。
- 在本项目中的作用：
  方向模型、情绪模型、风险模型、执行模型上线前后都需要挂接到模型注册表。

### 4.15 Market Regime / 市场状态

- 中文：市场状态、市场风格状态
- 英文：Market Regime
- 缩写：Regime
- 详细定义：
  Market Regime 描述某段时间内市场的典型行为模式，例如趋势市、震荡市、高波动期、低流动性期、风险偏好上升期等。
- 易混概念：
  市场状态不是单一因子，而是多种市场行为共同形成的环境标签。
- 在本项目中的作用：
  策略切换、参数自适应、回测分层分析和风险限制，都需要使用 Regime 识别结果。

## 5. 市场数据与微观结构术语

### 5.1 K 线 / Candlestick / Bar

- 中文：K 线、Bar
- 英文：Candlestick、Bar、K-line
- 缩写：Bar
- 详细定义：
  K 线是按固定时间窗口聚合后的行情数据，通常至少包含开盘价、最高价、最低价、收盘价和成交量，即 OHLCV。
- 易混概念：
  K 线是聚合数据，不等于真实逐笔成交路径。低频策略常用 K 线，高频和执行类策略通常需要更细粒度数据。
- 在本项目中的作用：
  MVP 阶段的基础回测、研究和多数策略开发，以 K 线数据作为核心输入。

### 5.2 Tick / 逐笔数据

- 中文：逐笔数据、Tick 数据
- 英文：Tick
- 缩写：Tick
- 详细定义：
  Tick 表示更细粒度的市场更新，可能是逐笔成交、逐笔报价、盘口变化或成交回报。它更接近真实市场状态。
- 易混概念：
  Tick 不一定代表一笔成交，也可能只是报价变化。不同数据供应商的 Tick 定义并不完全相同。
- 在本项目中的作用：
  Tick 数据用于高精度回测、订单簿因子、做市策略、执行分析和智能路由。

### 5.3 Quote / 报价

- 中文：报价
- 英文：Quote
- 缩写：无固定缩写
- 详细定义：
  Quote 指某一时刻的买卖报价信息，通常包含买价、卖价、对应挂单量以及时间戳。
- 易混概念：
  Quote 是报价意图，不等于实际成交。Trade 才是已经发生的成交事件。
- 在本项目中的作用：
  报价是估值、中间价、滑点估算、成交模拟和执行质量分析的重要基础数据。

### 5.4 Trade / 成交

- 中文：成交、逐笔成交
- 英文：Trade
- 缩写：无固定缩写
- 详细定义：
  Trade 是已经在市场中撮合完成的交易记录，通常包含成交价、成交量、成交方向和成交时间。
- 易混概念：
  Quote 体现挂单，Trade 体现成交。两者共同构成市场微观结构数据。
- 在本项目中的作用：
  成交流用于成交量分析、VWAP、订单簿不平衡研究、市场冲击评估和回测成交模拟。

### 5.5 Order Book / 订单簿

- 中文：订单簿
- 英文：Order Book
- 缩写：OB
- 详细定义：
  订单簿是市场当前各个价位上的挂单队列，包含买盘和卖盘深度。它反映了市场流动性、排队结构和短期供需关系。
- 易混概念：
  订单簿是动态结构，不是简单的静态深度快照。做市和执行策略更关注其动态变化。
- 在本项目中的作用：
  做市、智能执行、订单簿因子、冲击建模和微观结构回测都依赖订单簿。

### 5.6 L1 / L2 / L3 行情

- 中文：一级、二级、三级行情
- 英文：Level 1、Level 2、Level 3 Market Data
- 缩写：L1、L2、L3
- 详细定义：
  L1 通常指最优买卖价和最新成交等顶层数据；L2 通常指多档深度；L3 则可能包含订单级别的更细粒度队列信息。
- 易混概念：
  不同交易所和供应商对 L2、L3 的定义可能不同，接入时需要以接口文档为准。
- 在本项目中的作用：
  数据接入层需要统一封装不同档位行情，并在回测和执行服务里做能力降级处理。

### 5.7 Best Bid / Best Ask / 买一价与卖一价

- 中文：买一价、卖一价
- 英文：Best Bid、Best Ask
- 缩写：Bid、Ask
- 详细定义：
  Best Bid 是当前市场最高买价，Best Ask 是当前市场最低卖价。它们定义了最即时的成交边界。
- 易混概念：
  最优买卖价反映当前盘口，并不保证大额订单可以按该价格全部成交。
- 在本项目中的作用：
  做市报价、滑点估算、成交价模拟和风控检查都高度依赖最优买卖价。

### 5.8 Mid Price / 中间价

- 中文：中间价
- 英文：Mid Price
- 缩写：Mid
- 详细定义：
  中间价通常定义为最优买价与最优卖价的平均值，用于表示当前相对中性的参考价格。
- 易混概念：
  Mid Price 并不是一定能成交的价格，它更适合作为估值和分析参考。
- 在本项目中的作用：
  组合估值、做市报价、执行分析、微观结构因子经常使用中间价。

### 5.9 Spread / 点差

- 中文：点差、买卖价差
- 英文：Spread、Bid-Ask Spread
- 缩写：Spread
- 详细定义：
  点差是最优卖价与最优买价之间的差值，既可以用绝对价格表示，也可以用基点或百分比表示。
- 易混概念：
  成交量大并不必然意味着点差小。点差、深度和成交活跃度共同决定真实流动性。
- 在本项目中的作用：
  点差是流动性评价、交易成本估算、做市利润空间和执行质量分析的核心指标。

### 5.10 Slippage / 滑点

- 中文：滑点
- 英文：Slippage
- 缩写：无固定缩写
- 详细定义：
  滑点是订单预期成交价与实际成交价之间的偏差，来源包括点差、盘口深度不足、市场冲击、撮合延迟和价格跳变。
- 易混概念：
  滑点不只来自手续费，也不只是“价格差一点点”。在实盘中，滑点常常是策略收益失真的主要来源。
- 在本项目中的作用：
  回测引擎、模拟盘和实盘复盘都必须显式建模和记录滑点。

### 5.11 Market Impact / 市场冲击

- 中文：市场冲击
- 英文：Market Impact
- 缩写：无固定缩写
- 详细定义：
  市场冲击是指订单自身对市场价格和订单簿造成的影响，通常在大额交易或流动性不足时更明显。
- 易混概念：
  滑点是结果，市场冲击是产生滑点的一个重要原因，但两者不完全等同。
- 在本项目中的作用：
  智能执行算法、容量分析、大资金组合调仓和 TCA 需要单独评估市场冲击。

### 5.12 Liquidity / 流动性

- 中文：流动性
- 英文：Liquidity
- 缩写：无固定缩写
- 详细定义：
  流动性表示资产可以在较低成本和较小价格影响下被买入或卖出的能力。它与成交量、点差、深度、冲击成本和活跃度相关。
- 易混概念：
  成交量高不一定流动性好，尤其在盘口深度浅、波动剧烈时更是如此。
- 在本项目中的作用：
  Universe 筛选、仓位上限、执行算法选择和风控约束都需要依赖流动性评价。

### 5.13 Volatility / 波动率

- 中文：波动率
- 英文：Volatility
- 缩写：Vol
- 详细定义：
  波动率是价格波动强弱的度量，可以用历史收益波动计算，也可以从期权价格中反推出隐含波动率。
- 易混概念：
  波动率不等于风险的全部，但它是风险管理中最常用的代理指标之一。
- 在本项目中的作用：
  波动率用于仓位调整、止损止盈、波动率目标策略、风险预算和衍生品分析。

### 5.14 Open Interest / 未平仓量

- 中文：未平仓量
- 英文：Open Interest
- 缩写：OI
- 详细定义：
  未平仓量表示市场上尚未被对冲或平掉的合约总量，用于描述衍生品市场杠杆活跃程度。
- 易混概念：
  OI 不是成交量。成交量表示某段时间发生了多少交易，OI 表示当前仍存续多少合约。
- 在本项目中的作用：
  OI 是期货和加密衍生品策略、情绪分析和风险监控的重要输入。

### 5.15 Funding Rate / 资金费率

- 中文：资金费率
- 英文：Funding Rate
- 缩写：Funding
- 详细定义：
  资金费率是永续合约市场为使合约价格向现货价格收敛而设置的多空资金交换机制，通常按固定间隔结算。
- 易混概念：
  资金费率不是保证金利率，也不是借币利率。它是永续合约特有的价格锚定机制。
- 在本项目中的作用：
  资金费率是加密货币衍生品策略、套利策略、风控和 PnL 计算的重要字段。

### 5.16 Basis / 基差

- 中文：基差
- 英文：Basis
- 缩写：无固定缩写
- 详细定义：
  基差通常表示期货或永续价格与现货价格之间的差值，反映市场预期、期限结构和资金成本。
- 易混概念：
  基差不是单纯的“溢价”或“折价”标签，它通常需要结合期限、资金成本和交割机制解释。
- 在本项目中的作用：
  期货套利、跨期套利、现货对冲和方向判断都需要跟踪基差。

### 5.17 Term Structure / 期限结构

- 中文：期限结构
- 英文：Term Structure
- 缩写：无固定缩写
- 详细定义：
  期限结构描述同一标的不同到期合约之间的价格关系，例如升水结构、贴水结构和曲线陡峭程度。
- 易混概念：
  基差通常聚焦某一合约与现货的差，期限结构则关注多个不同到期合约之间的整体曲线关系。
- 在本项目中的作用：
  期限结构用于期货展期、跨期套利、库存策略和宏观商品分析。

### 5.18 Index Price / Mark Price / Last Price / 指数价、标记价、最新价

- 中文：指数价格、标记价格、最新成交价
- 英文：Index Price、Mark Price、Last Price
- 缩写：无固定缩写
- 详细定义：
  Index Price 是外部参考基准价格，Mark Price 是风控和估值参考价格，Last Price 是市场最近一笔成交价。三者在衍生品市场中用途不同。
- 易混概念：
  最新成交价最容易受短时成交影响；标记价格更适合风控；指数价格更适合做锚定参考。
- 在本项目中的作用：
  风控、持仓估值、强平监控、PnL 计算和情绪分析需要明确使用哪一种价格口径。

### 5.19 Corporate Actions / 公司行为

- 中文：公司行为
- 英文：Corporate Actions
- 缩写：无固定缩写
- 详细定义：
  公司行为包括分红、送股、拆股、并股、配股、增发、退市、并购等，会直接影响历史价格、股本、流通盘和持仓价值。
- 易混概念：
  公司行为调整不是简单的“复权价格处理”，还会影响回测、持仓、订单有效性和绩效解释。
- 在本项目中的作用：
  股票市场支持模块必须维护公司行为事件，并在研究、回测、实盘对账和报表中统一处理。

### 5.20 Trading Calendar / 交易日历

- 中文：交易日历
- 英文：Trading Calendar
- 缩写：无固定缩写
- 详细定义：
  交易日历描述交易所或市场的开市日、休市日、交易时段、午休、夜盘、盘前盘后、半日市和特殊安排。
- 易混概念：
  交易日历不只是一个节假日表，它还影响回测时间推进、数据对齐、订单接受时间和结算逻辑。
- 在本项目中的作用：
  A 股、港股、美股、期货和加密货币具有不同交易日历，平台必须统一管理并按市场差异执行。

## 6. 策略、回测与建模术语

### 6.1 Strategy / 策略

- 中文：策略
- 英文：Strategy
- 缩写：无固定缩写
- 详细定义：
  策略是一套把数据、规则、模型和风险约束转化为交易决策的完整逻辑，可以是规则型、统计型、机器学习型或执行型。
- 易混概念：
  单一因子、单个指标或单个模型不等于完整策略，策略还包括持仓、调仓、执行和风控。
- 在本项目中的作用：
  策略模块是平台的核心业务对象，需要支持版本化、参数化、回测、模拟盘和实盘部署。

### 6.2 Backtest / 回测

- 中文：回测
- 英文：Backtest
- 缩写：BT
- 详细定义：
  回测是用历史数据模拟策略在过去的运行表现，评估其收益、风险、稳健性和行为特征。
- 易混概念：
  回测结果不是未来实盘结果。历史重放无法完全复原真实成交、流动性和竞争环境。
- 在本项目中的作用：
  平台必须提供统一回测引擎，并显式建模滑点、手续费、资金费率、结算和公司行为。

### 6.3 Event-Driven Backtest / 事件驱动回测

- 中文：事件驱动回测
- 英文：Event-Driven Backtest
- 缩写：无固定缩写
- 详细定义：
  事件驱动回测以行情更新、订单回报、成交回报、定时器事件、新闻事件等为驱动源，按事件顺序推进系统状态。
- 易混概念：
  它比简单按 K 线循环的回测更接近真实交易系统，但实现复杂度也更高。
- 在本项目中的作用：
  事件驱动是本平台回测、模拟盘、实盘运行时统一化的重要设计原则。

### 6.4 Replay / 回放

- 中文：回放、事件回放
- 英文：Replay
- 缩写：无固定缩写
- 详细定义：
  Replay 指按历史事件原始顺序重新播放市场、订单和系统事件，用于复现实验、复盘故障和验证策略行为。
- 易混概念：
  Replay 不等于常规回测。回放更强调还原系统事件序列和状态变化，而不仅是计算收益曲线。
- 在本项目中的作用：
  回放用于故障复盘、执行质量分析、策略行为核验和 UAT 验收。

### 6.5 Paper Trading / 模拟盘

- 中文：模拟盘、纸面交易
- 英文：Paper Trading
- 缩写：Paper
- 详细定义：
  模拟盘是连接真实市场数据但不产生真实资金交割的交易环境，通常用于上线前验证策略逻辑和系统稳定性。
- 易混概念：
  模拟盘比回测更接近实时运行，但仍无法完全复刻真实成交队列、延迟和对手盘行为。
- 在本项目中的作用：
  这是实盘前的关键闸门，用于验证策略、风控、路由、监控和告警链路。

### 6.6 In-Sample / Out-of-Sample / 样本内与样本外

- 中文：样本内、样本外
- 英文：In-Sample、Out-of-Sample
- 缩写：IS、OOS
- 详细定义：
  样本内数据用于研究、拟合和参数选择，样本外数据用于验证模型或策略在未见数据上的表现。
- 易混概念：
  如果研究过程反复查看样本外结果并据此调整参数，样本外也会被“污染”。
- 在本项目中的作用：
  回测和实验追踪需要显式记录数据切分边界，避免评估失真。

### 6.7 Walk-Forward / 滚动验证

- 中文：滚动验证、步进式验证
- 英文：Walk-Forward Analysis
- 缩写：WFA
- 详细定义：
  Walk-Forward 是将历史按时间分段，循环进行“训练一段、验证一段”的评估方法，用于模拟策略在实际时间推进中的再训练与部署过程。
- 易混概念：
  它不是普通随机交叉验证，更适合时间序列和市场数据场景。
- 在本项目中的作用：
  机器学习策略、参数优化和 Regime 自适应策略应优先采用滚动验证。

### 6.8 Cross-Validation / 交叉验证

- 中文：交叉验证
- 英文：Cross-Validation
- 缩写：CV
- 详细定义：
  交叉验证是一种通过多次切分数据来评估模型泛化能力的方法。时间序列问题通常需要使用带时间顺序约束的版本。
- 易混概念：
  不能把普通随机打散的 K 折交叉验证直接用于时间序列，否则容易引入未来信息泄漏。
- 在本项目中的作用：
  用于模型策略的研究评估，但必须遵守时间因果顺序。

### 6.9 Look-Ahead Bias / 前视偏差

- 中文：前视偏差
- 英文：Look-Ahead Bias
- 缩写：无固定缩写
- 详细定义：
  前视偏差是指在历史回测或训练时使用了当时实际上还不可获得的未来信息，导致策略表现被高估。
- 易混概念：
  这类偏差往往不明显，可能来自数据发布日期、复权处理、财报滞后、标签构造和特征对齐。
- 在本项目中的作用：
  回测引擎、特征计算和测试用例必须专门防范前视偏差。

### 6.10 Survivor Bias / 幸存者偏差

- 中文：幸存者偏差
- 英文：Survivor Bias
- 缩写：无固定缩写
- 详细定义：
  幸存者偏差是指研究样本只保留当前仍存在的标的，而忽略了退市、停牌、清算或失去流动性的资产，从而高估策略历史表现。
- 易混概念：
  这在股票、基金和加密货币市场都存在，不只是股票回测问题。
- 在本项目中的作用：
  Universe、公司行为、退市事件和历史成分数据必须纳入回测体系。

### 6.11 Data Leakage / 数据泄漏

- 中文：数据泄漏
- 英文：Data Leakage
- 缩写：无固定缩写
- 详细定义：
  数据泄漏是指模型训练或评估阶段使用了不应知道的信息，导致离线表现虚高而线上失效。
- 易混概念：
  数据泄漏不仅是未来信息，还可能来自标签构造错误、归一化范围错误、跨样本污染和特征窗口错误。
- 在本项目中的作用：
  特征流水线、标签生成、训练脚本和测试用例必须有专门校验机制。

### 6.12 Overfitting / Underfitting / 过拟合与欠拟合

- 中文：过拟合、欠拟合
- 英文：Overfitting、Underfitting
- 缩写：无固定缩写
- 详细定义：
  过拟合表示模型过度记忆历史噪声，在新数据上失效；欠拟合表示模型过于简单，无法捕捉有效规律。
- 易混概念：
  复杂模型不必然更好，简单模型也不必然更稳。关键在于泛化能力和可重复性。
- 在本项目中的作用：
  研究与模型治理模块需要通过样本外验证、滚动验证和漂移监控来控制过拟合。

### 6.13 Drift / 漂移

- 中文：漂移
- 英文：Drift
- 缩写：无固定缩写
- 详细定义：
  Drift 通常指数据分布、特征统计性质、标签关系或模型表现随时间发生变化，导致训练时的规律不再稳定。
- 易混概念：
  漂移不一定意味着模型失效，但它是模型衰退的重要预警信号。
- 在本项目中的作用：
  模型治理模块需要监控数据漂移、概念漂移和性能漂移，并触发再训练或降级。

### 6.14 Regime Shift / 状态切换

- 中文：市场状态切换
- 英文：Regime Shift
- 缩写：无固定缩写
- 详细定义：
  Regime Shift 指市场从一种行为模式切换到另一种模式，例如从低波动切到高波动、从趋势行情切到震荡行情。
- 易混概念：
  Regime 是状态本身，Regime Shift 是状态变化事件。
- 在本项目中的作用：
  状态切换是策略失效、风控收紧、参数重估和信号置信度调整的重要触发器。

## 7. 订单、执行与交易术语

### 7.1 Order / 订单

- 中文：订单
- 英文：Order
- 缩写：无固定缩写
- 详细定义：
  订单是向交易场所提交的买卖指令，包含方向、价格、数量、订单类型、有效方式和附加限制。
- 易混概念：
  订单是执行对象，不是策略意图的全部表达。策略意图往往需要先转成目标仓位或执行计划。
- 在本项目中的作用：
  OMS 是订单生命周期管理中心，所有交易行为最终都需要落到订单实体上。

### 7.2 Position / 持仓

- 中文：持仓
- 英文：Position
- 缩写：Pos
- 详细定义：
  持仓表示账户或组合当前持有的资产数量、方向、成本和相关风险状态。
- 易混概念：
  持仓是当前结果，订单是改变持仓的手段，目标仓位是希望达到的未来状态。
- 在本项目中的作用：
  风控、估值、PnL、组合分析和调仓决策都建立在持仓对象之上。

### 7.3 Target Position / 目标仓位

- 中文：目标仓位
- 英文：Target Position
- 缩写：无固定缩写
- 详细定义：
  目标仓位表示策略和组合模块希望最终达到的资产暴露水平，通常由信号、权重、风险预算和约束共同决定。
- 易混概念：
  目标仓位不是订单；一个目标仓位可能需要多笔订单、多阶段执行和风控审查才能完成。
- 在本项目中的作用：
  平台推荐采用“信号 -> 目标仓位 -> 执行计划 -> 订单”的标准链路。

### 7.4 OMS / Order Management System / 订单管理系统

- 中文：订单管理系统
- 英文：Order Management System
- 缩写：OMS
- 详细定义：
  OMS 用于管理订单创建、校验、状态变更、撤单、重试、回报归档和审计链路，是所有订单生命周期的事实来源。
- 易混概念：
  OMS 更关注订单对象本身的完整生命周期，EMS 更关注如何执行得更好。
- 在本项目中的作用：
  OMS 是实盘与模拟盘中最关键的核心服务之一，需要与风控、账户、执行和监控紧密集成。

### 7.5 EMS / Execution Management System / 执行管理系统

- 中文：执行管理系统
- 英文：Execution Management System
- 缩写：EMS
- 详细定义：
  EMS 用于将目标仓位或母单转化为具体执行策略，包括拆单、排队、路由、成交跟踪、撤改逻辑和执行质量分析。
- 易混概念：
  EMS 不是简单的下单接口，而是“如何成交得更好”的系统。
- 在本项目中的作用：
  高流动性、多市场和大资金场景下，EMS 会直接决定策略能否把离线优势兑现为真实收益。

### 7.6 Market Order / 市价单

- 中文：市价单
- 英文：Market Order
- 缩写：无固定缩写
- 详细定义：
  市价单以市场当前可成交的最优价格立即成交，优先保证成交速度而不是成交价格。
- 易混概念：
  市价单不保证成交价稳定，在流动性差或波动剧烈时可能产生较大滑点。
- 在本项目中的作用：
  市价单适用于紧急止损、快速建仓或对价格不敏感的场景，但必须受到风控限制。

### 7.7 Limit Order / 限价单

- 中文：限价单
- 英文：Limit Order
- 缩写：无固定缩写
- 详细定义：
  限价单指定可接受的最差成交价格，优先控制价格，但不保证一定成交。
- 易混概念：
  限价单不一定更“安全”。如果价格设置过于激进，可能长期排队或错失成交机会。
- 在本项目中的作用：
  限价单是做市、被动成交、成本控制和智能执行的基础订单类型。

### 7.8 Stop Order / Stop-Limit Order / 止损单与止损限价单

- 中文：止损单、止损限价单
- 英文：Stop Order、Stop-Limit Order
- 缩写：无固定缩写
- 详细定义：
  止损单在触发价被击穿时转为市价单；止损限价单在触发后转为限价单。它们用于风险控制和突破跟随。
- 易混概念：
  止损限价单虽然价格更可控，但在剧烈波动时更容易触发后无法成交。
- 在本项目中的作用：
  股票、期货和加密货币策略都可能使用止损类订单，但要根据交易所支持情况做适配。

### 7.9 Time in Force / 订单有效方式

- 中文：订单有效方式
- 英文：Time in Force
- 缩写：TIF
- 详细定义：
  TIF 用于定义订单在市场中保持有效的时间规则，例如撤销前有效、立即成交否则取消等。
- 易混概念：
  TIF 不是订单类型，而是订单属性。
- 在本项目中的作用：
  OMS 和适配器必须标准化 TIF，避免不同交易所口径不一致。

### 7.10 GTC / 撤销前有效

- 中文：撤销前有效
- 英文：Good Till Cancelled
- 缩写：GTC
- 详细定义：
  GTC 订单在未成交或未被主动撤销前持续有效，具体有效期可能受交易所或券商限制。
- 易混概念：
  并非所有市场都真正支持无限期 GTC，有些会在日终或一段时间后强制失效。
- 在本项目中的作用：
  适合低频调仓、挂单等待和做市，但需要与交易日历和会话规则联动。

### 7.11 IOC / 立即成交否则取消

- 中文：立即成交否则取消
- 英文：Immediate or Cancel
- 缩写：IOC
- 详细定义：
  IOC 订单会立即尝试成交，可部分成交，剩余未成交部分立即取消。
- 易混概念：
  IOC 可以部分成交，FOK 不可以。
- 在本项目中的作用：
  适用于流动性试探、被动风控和快速执行。

### 7.12 FOK / 全成否则取消

- 中文：全成否则取消
- 英文：Fill or Kill
- 缩写：FOK
- 详细定义：
  FOK 要求订单必须立即全部成交，否则整笔订单取消。
- 易混概念：
  FOK 更强调成交完整性，通常用于对碎片成交非常敏感的场景。
- 在本项目中的作用：
  在套利、跨腿交易或对冲腿同步性要求较高时更有价值。

### 7.13 Post-Only / 只挂单

- 中文：只挂单
- 英文：Post-Only
- 缩写：无固定缩写
- 详细定义：
  Post-Only 要求订单只能以挂单身份进入订单簿，如果会立即吃单成交，则订单会被拒绝或自动调整。
- 易混概念：
  Post-Only 不是限价单的同义词，它是在限价单基础上进一步限制成交方式。
- 在本项目中的作用：
  做市、手续费优化和被动流动性策略常使用 Post-Only。

### 7.14 Reduce-Only / 只减仓

- 中文：只减仓
- 英文：Reduce-Only
- 缩写：无固定缩写
- 详细定义：
  Reduce-Only 订单只允许减少现有持仓，不允许因误操作而开出更大反向仓位。
- 易混概念：
  它是风控保护属性，而不是独立订单类型。
- 在本项目中的作用：
  合约和杠杆市场的平仓、止盈、止损逻辑应优先支持 Reduce-Only。

### 7.15 OCO / 二选一订单

- 中文：二选一订单
- 英文：One Cancels the Other
- 缩写：OCO
- 详细定义：
  OCO 由两个相互关联的订单组成，其中一个成交或触发后，另一个自动取消。
- 易混概念：
  OCO 不是所有交易场所的原生能力，有时需要在平台侧模拟实现。
- 在本项目中的作用：
  常用于止盈止损成对设置和突破交易。

### 7.16 Iceberg Order / 冰山订单

- 中文：冰山订单
- 英文：Iceberg Order
- 缩写：无固定缩写
- 详细定义：
  冰山订单把大额订单拆成可见部分和隐藏部分，减少市场察觉和价格冲击。
- 易混概念：
  冰山订单不等于普通拆单，它强调在交易所或平台层面保持部分隐藏展示。
- 在本项目中的作用：
  大额调仓、低冲击执行和做市保护场景有较高价值。

### 7.17 TWAP / 时间加权平均价格算法

- 中文：时间加权平均价格算法
- 英文：Time Weighted Average Price
- 缩写：TWAP
- 详细定义：
  TWAP 按时间均匀分布订单执行量，目标是在一定时间窗口内将平均成交价格接近时间均值。
- 易混概念：
  TWAP 不考虑市场成交量分布，适合对执行节奏可控但不强调跟随市场成交量的场景。
- 在本项目中的作用：
  EMS 需要支持 TWAP 作为基础执行算法之一。

### 7.18 VWAP / 成交量加权平均价格算法

- 中文：成交量加权平均价格算法
- 英文：Volume Weighted Average Price
- 缩写：VWAP
- 详细定义：
  VWAP 以市场成交量分布为参考，将订单量安排在预期成交活跃时段，以使成交均价接近市场成交量加权均价。
- 易混概念：
  VWAP 需要对市场成交量分布有估计，难度高于 TWAP。
- 在本项目中的作用：
  股票、期货等流动性较稳定市场常使用 VWAP 作为执行基准和算法。

### 7.19 POV / 成交量参与率算法

- 中文：成交量参与率算法
- 英文：Percentage of Volume
- 缩写：POV
- 详细定义：
  POV 通过限制订单参与市场成交量的比例来控制执行速度和冲击成本。
- 易混概念：
  POV 是对“参与率”的约束，不一定以固定时间或固定份额完成。
- 在本项目中的作用：
  大额订单执行和流动性敏感场景适合使用 POV。

### 7.20 Smart Order Router / 智能订单路由

- 中文：智能订单路由
- 英文：Smart Order Router
- 缩写：SOR
- 详细定义：
  智能订单路由根据价格、深度、延迟、费用、成交概率、库存和风险约束，在多个交易场所之间选择最优下单路径。
- 易混概念：
  SOR 不只是比价，它还需要综合考虑成交质量、合规边界和执行风险。
- 在本项目中的作用：
  多交易所加密货币、多券商股票和多市场执行场景都需要 SOR。

### 7.21 Venue / 交易场所

- 中文：交易场所、成交场所
- 英文：Venue
- 缩写：无固定缩写
- 详细定义：
  Venue 指订单可以送达并成交的具体场所，例如交易所、券商内部化渠道、OTC 柜台或 DEX。
- 易混概念：
  Venue 比 Exchange 更宽泛，因为它不一定是公开交易所。
- 在本项目中的作用：
  适配器、路由策略、费用模型和监控报表都要按 Venue 维度管理。

### 7.22 Fill Rate / 成交率

- 中文：成交率
- 英文：Fill Rate
- 缩写：无固定缩写
- 详细定义：
  成交率用于描述订单提交后实际成交的比例，可按数量、金额、时间窗口或策略类别统计。
- 易混概念：
  成交率高不一定代表执行质量高，还需要结合价格偏离、滑点和市场冲击分析。
- 在本项目中的作用：
  执行评估和 TCA 模块需要持续跟踪成交率。

## 8. 投资组合、风险与绩效术语

### 8.1 Portfolio / 投资组合

- 中文：投资组合
- 英文：Portfolio
- 缩写：PF
- 详细定义：
  投资组合是按统一目标、约束和风险预算管理的一组持仓集合。它可以是一只策略、一个账户或一组子策略的聚合体。
- 易混概念：
  账户不一定等于组合。一个账户中可以运行多个组合，一个组合也可能跨账户部署。
- 在本项目中的作用：
  组合服务负责目标权重、资金分配、暴露控制、再平衡和组合绩效分析。

### 8.2 Weight / Allocation / 权重与资金分配

- 中文：权重、资金分配
- 英文：Weight、Allocation
- 缩写：无固定缩写
- 详细定义：
  权重表示某资产在组合中的相对占比，Allocation 更强调整体资金或风险预算如何分配给不同策略、资产或市场。
- 易混概念：
  权重可以按市值、风险、保证金占用或名义金额定义，必须明确口径。
- 在本项目中的作用：
  目标仓位、组合约束和再平衡都围绕权重和分配逻辑展开。

### 8.3 Gross Exposure / 总敞口

- 中文：总敞口
- 英文：Gross Exposure
- 缩写：无固定缩写
- 详细定义：
  总敞口通常是组合中多头名义敞口与空头名义敞口绝对值之和，反映整体风险使用强度。
- 易混概念：
  总敞口不考虑多空抵消方向，净敞口才考虑方向。
- 在本项目中的作用：
  杠杆组合、市场中性组合和多策略组合都需要限制总敞口。

### 8.4 Net Exposure / 净敞口

- 中文：净敞口
- 英文：Net Exposure
- 缩写：无固定缩写
- 详细定义：
  净敞口是多头敞口与空头敞口相抵之后的方向性暴露，反映组合对市场整体涨跌的敏感性。
- 易混概念：
  净敞口接近零不代表风险很小，若总敞口很大，组合仍可能有较高波动和执行风险。
- 在本项目中的作用：
  市场中性、对冲和风格中性策略需要重点监控净敞口。

### 8.5 Notional / 名义价值

- 中文：名义价值
- 英文：Notional Value
- 缩写：Notional
- 详细定义：
  名义价值是持仓或订单按价格和合约乘数换算后的总金额，用于统一比较不同资产和不同合约规模。
- 易混概念：
  名义价值不等于实际占用现金，杠杆和保证金产品里两者区别很大。
- 在本项目中的作用：
  风控、仓位上限、敞口统计和跨市场统一报表都需要名义价值口径。

### 8.6 Leverage / 杠杆

- 中文：杠杆
- 英文：Leverage
- 缩写：Lev
- 详细定义：
  杠杆表示实际承担的风险敞口相对净资产的放大倍数，可以按名义敞口、保证金占用或风险暴露衡量。
- 易混概念：
  杠杆高不必然收益高，但会显著放大波动、回撤和强平风险。
- 在本项目中的作用：
  杠杆是组合和账户级风控的重要约束参数。

### 8.7 Margin / 保证金

- 中文：保证金
- 英文：Margin
- 缩写：无固定缩写
- 详细定义：
  保证金是投资者为维持杠杆头寸而需要提供的担保资金，常见于期货、期权、融资融券和加密货币合约。
- 易混概念：
  保证金不是最大损失的上限，价格剧烈波动时损失可能大于初始保证金。
- 在本项目中的作用：
  平台必须实时跟踪保证金率、可用保证金和风险等级。

### 8.8 Initial Margin / Maintenance Margin / 初始保证金与维持保证金

- 中文：初始保证金、维持保证金
- 英文：Initial Margin、Maintenance Margin
- 缩写：IM、MM
- 详细定义：
  初始保证金是开仓所需的最低保证金，维持保证金是持仓期间必须保持的最低保证金水平，低于维持保证金可能触发追加保证金或强平。
- 易混概念：
  两者不是同一个数值，维持保证金通常低于初始保证金，但不同市场规则不同。
- 在本项目中的作用：
  期货和加密货币衍生品风控需要区分 IM 和 MM。

### 8.9 Mark-to-Market / 盯市

- 中文：盯市、按市值计价
- 英文：Mark-to-Market
- 缩写：MTM
- 详细定义：
  盯市是按当前市场价格或结算价格重新计算持仓价值与盈亏的过程。期货市场通常以日结算价进行盯市。
- 易混概念：
  盯市不等于真实平仓，但会改变浮动盈亏、保证金和风险状态。
- 在本项目中的作用：
  组合估值、保证金管理、日终结算和实盘报表都必须支持盯市。

### 8.10 PnL / 盈亏

- 中文：盈亏
- 英文：Profit and Loss
- 缩写：PnL
- 详细定义：
  PnL 表示交易、持仓、组合或账户的损益结果，可分为已实现盈亏和未实现盈亏。
- 易混概念：
  PnL 必须明确是否包含手续费、资金费率、借贷成本、滑点和结算调整。
- 在本项目中的作用：
  回测、模拟盘、实盘、风控、报表与复盘模块都围绕 PnL 展开。

### 8.11 NAV / AUM / 净值与管理规模

- 中文：净值、管理资产规模
- 英文：Net Asset Value、Assets Under Management
- 缩写：NAV、AUM
- 详细定义：
  NAV 表示组合或账户的净资产价值，AUM 表示管理中的总资产规模。两者经常共同用于描述策略体量和成长情况。
- 易混概念：
  NAV 更像时点状态，AUM 更像规模概念。不同机构对 AUM 的口径可能不完全一致。
- 在本项目中的作用：
  组合报表、账户总览和绩效归因需要统一展示 NAV 和 AUM。

### 8.12 VaR / 风险价值

- 中文：风险价值
- 英文：Value at Risk
- 缩写：VaR
- 详细定义：
  VaR 表示在给定置信水平和持有期下，组合可能遭受的最大损失估计值。例如 99% 置信水平下一天 VaR。
- 易混概念：
  VaR 不是最坏损失，它只是在一定置信水平下的阈值，不描述阈值之外损失有多大。
- 在本项目中的作用：
  组合层和账户层风险监控可以将 VaR 作为核心参考指标之一。

### 8.13 CVaR / 条件风险价值

- 中文：条件风险价值、预期尾部损失
- 英文：Conditional Value at Risk
- 缩写：CVaR
- 详细定义：
  CVaR 衡量在损失已经超过 VaR 阈值的条件下，尾部损失的平均水平，因此更能反映极端风险。
- 易混概念：
  VaR 关注阈值位置，CVaR 关注阈值之外的尾部厚度。
- 在本项目中的作用：
  压力测试、极端风险控制和高杠杆场景应优先关注 CVaR。

### 8.14 Stress Test / 压力测试

- 中文：压力测试
- 英文：Stress Test
- 缩写：无固定缩写
- 详细定义：
  压力测试是在极端但合理的市场情境下评估组合损失、流动性压力和系统稳定性的过程。
- 易混概念：
  压力测试不只是把波动率放大，还应考虑跳空、熔断、流动性消失、相关性失效和系统故障。
- 在本项目中的作用：
  组合上线前和高风险时段应定期进行压力测试。

### 8.15 Drawdown / Max Drawdown / 回撤与最大回撤

- 中文：回撤、最大回撤
- 英文：Drawdown、Maximum Drawdown
- 缩写：DD、MDD
- 详细定义：
  回撤表示净值从历史峰值回落的幅度，最大回撤是历史观察期内出现过的最大回撤。
- 易混概念：
  高收益策略不一定是好策略，如果回撤不可承受，策略在实际运营中很难持续。
- 在本项目中的作用：
  回撤是风控阈值、策略停用条件和绩效报告的核心指标。

### 8.16 Sharpe Ratio / 夏普比率

- 中文：夏普比率
- 英文：Sharpe Ratio
- 缩写：Sharpe
- 详细定义：
  夏普比率衡量单位总波动风险所获得的超额收益，常用于评价策略风险调整后收益表现。
- 易混概念：
  夏普对收益分布的偏态和尾部并不敏感，因此不能单独作为唯一评价标准。
- 在本项目中的作用：
  回测、实盘评估和策略筛选常使用夏普比率作为一级指标。

### 8.17 Sortino Ratio / 索提诺比率

- 中文：索提诺比率
- 英文：Sortino Ratio
- 缩写：Sortino
- 详细定义：
  Sortino Ratio 用下行波动替代总波动，更强调“坏波动”带来的风险。
- 易混概念：
  当收益分布存在明显非对称性时，Sortino 往往比 Sharpe 更有解释力。
- 在本项目中的作用：
  对回撤敏感或收益分布偏斜的策略应重点参考 Sortino。

### 8.18 Calmar Ratio / 卡玛比率

- 中文：卡玛比率
- 英文：Calmar Ratio
- 缩写：Calmar
- 详细定义：
  Calmar Ratio 用年化收益除以最大回撤，强调收益与深度回撤之间的平衡。
- 易混概念：
  Calmar 更依赖回撤表现，因此对低频趋势或存在长回撤周期的策略很有参考意义。
- 在本项目中的作用：
  适用于趋势、CTA、跨周期组合等强调回撤控制的策略评价。

### 8.19 Information Ratio / 信息比率

- 中文：信息比率
- 英文：Information Ratio
- 缩写：IR
- 详细定义：
  信息比率是超额收益与跟踪误差的比值，用于衡量策略相对基准创造超额收益的稳定性。
- 易混概念：
  信息比率以基准为前提，因此必须先明确 Benchmark 才有意义。
- 在本项目中的作用：
  股票相对收益策略和组合管理模块需要使用 IR。

### 8.20 Turnover / 换手率

- 中文：换手率、组合换手
- 英文：Turnover
- 缩写：无固定缩写
- 详细定义：
  换手率表示组合或策略在一段时间内持仓变动的频繁程度，可按成交金额、仓位变化或名义价值计算。
- 易混概念：
  高换手不一定是坏事，但它通常意味着更高的滑点、费用和执行复杂度。
- 在本项目中的作用：
  换手率是评估策略可实施性和交易成本的重要指标。

### 8.21 Capacity / 容量

- 中文：策略容量
- 英文：Capacity
- 缩写：无固定缩写
- 详细定义：
  容量表示在不显著恶化收益、成本和风险特征的前提下，策略可承载的资金规模上限。
- 易混概念：
  回测容量往往会被高估，实盘容量必须考虑深度、冲击、路由和市场竞争。
- 在本项目中的作用：
  容量分析决定策略是否可扩大资金规模和是否适合多账户复制。

### 8.22 Kill Switch / 紧急停机开关

- 中文：紧急停机开关
- 英文：Kill Switch
- 缩写：无固定缩写
- 详细定义：
  Kill Switch 是在出现异常成交、风控失控、系统错误或市场极端事件时，立即停止新订单并进入保护状态的机制。
- 易混概念：
  Kill Switch 不只是一个按钮，它还需要配套触发条件、权限控制、回滚策略和恢复流程。
- 在本项目中的作用：
  这是实盘系统必须具备的最后一道安全防线。

## 9. 股票市场术语

### 9.1 Equities / 股票

- 中文：股票、权益类资产
- 英文：Equities
- 缩写：EQ
- 详细定义：
  股票是代表公司所有权份额的权益类证券，通常具有投票权、分红权和资本利得属性。
- 易混概念：
  股票市场的交易规则受市场区域影响很大，A 股、港股、美股在制度和接口上差异明显。
- 在本项目中的作用：
  平台需要统一支持 A 股、港股、美股的交易日历、公司行为、风控和券商适配。

### 9.2 A Shares / Hong Kong Stocks / US Stocks / A股、港股、美股

- 中文：A股、港股、美股
- 英文：A Shares、Hong Kong Stocks、US Stocks
- 缩写：CN、HK、US
- 详细定义：
  A 股通常指中国内地交易所上市股票；港股指香港交易所上市股票；美股指美国市场上市股票。
- 易混概念：
  三者在交易时段、涨跌停、结算周期、卖空规则、盘前盘后、最小报价单位和公司行为处理上差异显著。
- 在本项目中的作用：
  股票支持设计文档明确了这三类市场必须被分别建模，而不是简单用一个统一字段替代。

### 9.3 Lot Size / 交易单位

- 中文：交易单位、手数
- 英文：Lot Size
- 缩写：无固定缩写
- 详细定义：
  Lot Size 是交易最小单位或标准单位，不同市场对最小买卖单位的要求不同。
- 易混概念：
  A 股、港股、美股在整手、碎股和可交易最小单位上存在差异，不能混用。
- 在本项目中的作用：
  下单前检查、订单数量归整和回测撮合必须考虑 Lot Size。

### 9.4 Auction / 集合竞价

- 中文：集合竞价
- 英文：Auction、Call Auction
- 缩写：无固定缩写
- 详细定义：
  集合竞价是市场在特定时段统一撮合成交的交易机制，常出现在开盘、收盘或特殊时段。
- 易混概念：
  集合竞价和连续竞价的撮合逻辑不同，订单可接受性和成交规则也不同。
- 在本项目中的作用：
  股票和部分期货市场都需要在交易日历和执行引擎中支持集合竞价时段。

### 9.5 T+1 / T+2 / 结算周期

- 中文：T+1、T+2 结算
- 英文：T+1 Settlement、T+2 Settlement
- 缩写：无固定缩写
- 详细定义：
  结算周期指交易日之后若干工作日完成资金和证券交收的规则，例如 T+1、T+2。
- 易混概念：
  结算周期不一定等于是否允许当日回转交易，不同市场在“交易”和“结算”层面可能分开规定。
- 在本项目中的作用：
  账户、可用资金、可卖数量和对账模块必须纳入结算周期。

### 9.6 Halt / Suspension / 停牌与暂停交易

- 中文：停牌、暂停交易
- 英文：Trading Halt、Suspension
- 缩写：无固定缩写
- 详细定义：
  停牌或暂停交易表示某只证券在特定时间内不能正常交易，原因可能包括公告披露、异常波动、重大事项或监管措施。
- 易混概念：
  停牌不只是行情缺失，它会影响订单有效性、估值和风险暴露处理。
- 在本项目中的作用：
  股票市场数据层和风控层必须显式处理停牌标的。

### 9.7 Ex-Dividend / Ex-Rights / 除息与除权

- 中文：除息、除权
- 英文：Ex-Dividend、Ex-Rights
- 缩写：XD、XR
- 详细定义：
  除息和除权是公司行为生效后，股价和持仓价值进行相应调整的时点和机制。
- 易混概念：
  如果回测和实盘对除权除息处理不一致，会导致收益和信号严重失真。
- 在本项目中的作用：
  股票回测、净值计算和报表必须统一使用公司行为口径。

### 9.8 Short Selling / 卖空

- 中文：卖空、融券卖出
- 英文：Short Selling
- 缩写：Short
- 详细定义：
  卖空是先借入证券卖出，再在未来买回归还的一种交易方式，通常受市场规则和可借券源限制。
- 易混概念：
  做空能力不是所有股票市场和所有标的都天然具备，股票市场尤其需要检查可融券性。
- 在本项目中的作用：
  股票市场的策略模板和风控模块必须根据市场区域区分是否允许卖空。

### 9.9 Pattern Day Trader / 日内频繁交易账户

- 中文：日内频繁交易账户规则
- 英文：Pattern Day Trader
- 缩写：PDT
- 详细定义：
  PDT 是美国市场中的监管术语，用于描述在一定期限内频繁进行日内交易的保证金账户，并附带最低资产要求。
- 易混概念：
  这是典型的区域性合规术语，并非全球股票市场通用规则。
- 在本项目中的作用：
  美股市场支持和风控规则文档应保留 PDT 相关约束位。

### 9.10 Uptick Rule / 卖空上跳规则

- 中文：卖空上跳规则
- 英文：Uptick Rule
- 缩写：无固定缩写
- 详细定义：
  Uptick Rule 是限制卖空执行条件的规则，通常要求卖空在某些价格条件满足时才能进行。
- 易混概念：
  卖空规则可能按市场、券商、监管阶段和个股状态变化，不能静态假设。
- 在本项目中的作用：
  股票适配器和订单前风控需要保留按市场区域应用卖空规则的能力。

## 10. 期货、加密货币、期权与 DEX 术语

### 10.1 Futures / 期货

- 中文：期货
- 英文：Futures
- 缩写：FU
- 详细定义：
  期货是约定在未来特定时间以约定方式交割某一标的的标准化合约，常带有保证金和日盯市机制。
- 易混概念：
  期货不是永续合约。期货有明确到期和交割安排，永续合约通常没有固定到期日。
- 在本项目中的作用：
  平台需要支持期货的合约元数据、交割、结算、展期和连续合约映射。

### 10.2 Perpetual Futures / 永续合约

- 中文：永续合约
- 英文：Perpetual Futures、Perpetual Swap
- 缩写：Perp
- 详细定义：
  永续合约是一种没有固定到期日的衍生品合约，通常通过资金费率机制使价格向现货锚定。
- 易混概念：
  永续合约虽类似期货，但核心定价和风险控制口径更依赖资金费率和标记价格。
- 在本项目中的作用：
  加密货币模块必须原生支持永续合约的资金费率、强平、杠杆和风险监控。

### 10.3 Contract Multiplier / 合约乘数

- 中文：合约乘数
- 英文：Contract Multiplier
- 缩写：无固定缩写
- 详细定义：
  合约乘数是每张合约对应的标的数量或金额换算系数，用于将价格变化转换为真实盈亏和名义价值。
- 易混概念：
  合约张数不能直接跨品种比较，必须结合乘数和报价单位。
- 在本项目中的作用：
  期货和期权的下单、估值、风控和报表都必须使用合约乘数。

### 10.4 Settlement Price / 结算价

- 中文：结算价
- 英文：Settlement Price
- 缩写：无固定缩写
- 详细定义：
  结算价是交易所或场所用于日终盯市、盈亏结转和保证金计算的标准价格。
- 易混概念：
  结算价不一定等于收盘价，也不一定等于最后一笔成交价。
- 在本项目中的作用：
  期货持仓估值、日终结算、风险复盘和连续合约处理依赖结算价。

### 10.5 Delivery / Expiry / 交割与到期

- 中文：交割、到期
- 英文：Delivery、Expiry
- 缩写：无固定缩写
- 详细定义：
  到期表示合约停止交易的时间点，交割表示合约到期后的履约与现金或实物结算过程。
- 易混概念：
  到期不代表自动关闭所有逻辑，交割还可能带来持仓迁移、现金流结算和报表处理。
- 在本项目中的作用：
  期货和期权模块必须跟踪到期和交割事件。

### 10.6 Rollover / 展期

- 中文：展期、移仓换月
- 英文：Rollover
- 缩写：Roll
- 详细定义：
  展期是将临近到期的合约头寸转移到更远月合约的过程，以维持策略连续暴露。
- 易混概念：
  展期不仅是平近月开远月，还包含时间选择、成本控制和流动性评估。
- 在本项目中的作用：
  CTA、商品、股指期货和连续合约研究都需要展期逻辑。

### 10.7 Continuous Contract / 连续合约

- 中文：连续合约
- 英文：Continuous Contract
- 缩写：无固定缩写
- 详细定义：
  连续合约是将不同到期合约按规则拼接形成的研究序列，用于画长期价格曲线、计算因子和回测策略。
- 易混概念：
  连续合约是研究数据，不是真实可交易单一合约，回测时需要处理移仓和跳价问题。
- 在本项目中的作用：
  期货研究和因子计算需要连续合约映射表。

### 10.8 Basis Trade / Carry / 基差交易与持有收益

- 中文：基差交易、Carry
- 英文：Basis Trade、Carry
- 缩写：无固定缩写
- 详细定义：
  基差交易利用现货与衍生品之间的价差关系获利，Carry 则更广义地表示持有资产或头寸所带来的结构性收益来源。
- 易混概念：
  Carry 不一定是无风险收益，它可能伴随流动性、资金、交割和市场结构风险。
- 在本项目中的作用：
  期货、永续、现货对冲和跨期策略设计中，Carry 是核心收益框架之一。

### 10.9 Liquidation / 强平

- 中文：强制平仓、强平
- 英文：Liquidation
- 缩写：无固定缩写
- 详细定义：
  强平是当账户保证金不足、风险率过高或违反场所规则时，交易所或券商自动关闭部分或全部持仓的机制。
- 易混概念：
  强平是风控结果，不是正常平仓方式。它通常意味着较差的执行价格和更高的损失风险。
- 在本项目中的作用：
  风控和监控系统必须对强平距离和强平事件进行实时预警。

### 10.10 Auto-Deleveraging / 自动减仓

- 中文：自动减仓
- 英文：Auto-Deleveraging
- 缩写：ADL
- 详细定义：
  ADL 是部分加密衍生品平台在极端风险下采用的机制，用于在保险基金不足时对盈利对手方进行强制减仓。
- 易混概念：
  ADL 与普通强平不同，它可能影响并未违规的一侧持仓。
- 在本项目中的作用：
  加密衍生品适配器和风控报表需要记录 ADL 风险信号。

### 10.11 Maker / Taker / 挂单方与吃单方

- 中文：挂单方、吃单方
- 英文：Maker、Taker
- 缩写：无固定缩写
- 详细定义：
  Maker 是提供流动性的一方，订单先进入订单簿等待成交；Taker 是消耗流动性的一方，订单立即与现有挂单成交。
- 易混概念：
  Maker/Taker 不只是手续费分类，也反映执行风格和市场冲击特征。
- 在本项目中的作用：
  执行算法、费用模型和做市策略需要精确区分 Maker 与 Taker 成本。

### 10.12 Borrow Rate / 借贷利率

- 中文：借贷利率、借币利率
- 英文：Borrow Rate
- 缩写：无固定缩写
- 详细定义：
  Borrow Rate 是融资融券或保证金借贷中，为借入现金或资产所需支付的利率或费用。
- 易混概念：
  Borrow Rate 与 Funding Rate 不同，一个对应借贷成本，一个对应永续合约多空资金交换。
- 在本项目中的作用：
  股票卖空、加密货币杠杆和套利策略都需要将借贷成本计入真实收益。

### 10.13 On-Chain Data / 链上数据

- 中文：链上数据
- 英文：On-Chain Data
- 缩写：无固定缩写
- 详细定义：
  链上数据是直接来自区块链账本和智能合约状态的数据，例如地址活跃度、转账规模、交易手续费、资金流向和协议 TVL。
- 易混概念：
  链上数据不是交易所行情数据，两者反映的是不同层面的市场行为。
- 在本项目中的作用：
  链上数据是加密货币另类数据因子的重要来源。

### 10.14 AMM / 自动做市商

- 中文：自动做市商
- 英文：Automated Market Maker
- 缩写：AMM
- 详细定义：
  AMM 是 DEX 中通过算法和流动性池自动给出报价并完成交易撮合的机制，不依赖传统订单簿。
- 易混概念：
  AMM 与中心化交易所订单簿机制不同，价格形成和流动性成本结构也不同。
- 在本项目中的作用：
  DEX 模块、链上执行和 LP 风险分析都需要理解 AMM 机制。

### 10.15 CLMM / 集中流动性做市

- 中文：集中流动性做市
- 英文：Concentrated Liquidity Market Maker
- 缩写：CLMM
- 详细定义：
  CLMM 允许流动性提供者把资金集中部署在选定价格区间，以提高资金效率并增加区间内做市收益。
- 易混概念：
  它比传统 AMM 复杂得多，因为 LP 同时承担区间选择和再平衡风险。
- 在本项目中的作用：
  DEX 做市模块需要支持 CLMM 头寸状态、区间风险和收益分析。

### 10.16 Liquidity Provider / 流动性提供者

- 中文：流动性提供者
- 英文：Liquidity Provider
- 缩写：LP
- 详细定义：
  LP 是向市场或流动性池提供买卖深度的参与者，可以是中心化交易所做市账户，也可以是 DEX 池子的流动性注入者。
- 易混概念：
  LP 不一定是传统意义上的做市商，但承担的流动性与库存风险相似。
- 在本项目中的作用：
  做市和 DEX 模块需要统一描述 LP 的头寸、库存、收益和风险。

### 10.17 Impermanent Loss / 无常损失

- 中文：无常损失
- 英文：Impermanent Loss
- 缩写：IL
- 详细定义：
  无常损失是 DEX 流动性提供者相对于单纯持有基础资产而言，因为价格变动和池子再平衡而出现的相对损失。
- 易混概念：
  无常损失并不总是“暂时”的，一旦退出流动性池，它就会变成已实现结果。
- 在本项目中的作用：
  DEX 做市和 LP 收益分析必须单独核算无常损失。

### 10.18 Option Chain / 期权链

- 中文：期权链
- 英文：Option Chain
- 缩写：无固定缩写
- 详细定义：
  期权链是按不同到期日和执行价组织的期权合约集合，用于观察市场隐含波动率、偏度和流动性分布。
- 易混概念：
  期权链不是单一合约，它是期权市场结构的全景视图。
- 在本项目中的作用：
  期权研究、波动率交易和风险对冲模块都依赖期权链数据。

### 10.19 Call / Put / 看涨期权与看跌期权

- 中文：看涨期权、看跌期权
- 英文：Call、Put
- 缩写：无固定缩写
- 详细定义：
  Call 赋予持有人在未来以约定价格买入标的的权利，Put 赋予持有人以约定价格卖出标的的权利。
- 易混概念：
  权利与义务是期权理解的核心。买方拥有权利，卖方承担义务。
- 在本项目中的作用：
  期权模块的合约建模、持仓分析和 Greeks 计算必须明确区分 Call 与 Put。

### 10.20 Strike Price / 执行价

- 中文：执行价
- 英文：Strike Price
- 缩写：Strike
- 详细定义：
  执行价是期权在行权时买卖标的的约定价格，是期权定价和盈亏结构的核心参数之一。
- 易混概念：
  执行价不是当前市场价。期权价值取决于执行价与标的现价、到期时间和波动率的共同关系。
- 在本项目中的作用：
  期权合约主数据和风险模型都依赖执行价。

### 10.21 Greeks / 希腊值

- 中文：希腊值
- 英文：Greeks
- 缩写：无固定缩写
- 详细定义：
  Greeks 是度量期权价格对不同风险因子敏感度的一组指标，常见包括 Delta、Gamma、Vega、Theta、Rho。
- 易混概念：
  Greeks 不是单一指标，它们分别描述方向风险、曲率风险、波动率风险、时间衰减和利率敏感度。
- 在本项目中的作用：
  期权风控、波动率交易和 Delta 中性策略需要持续计算 Greeks。

### 10.22 Implied Volatility / 隐含波动率

- 中文：隐含波动率
- 英文：Implied Volatility
- 缩写：IV
- 详细定义：
  隐含波动率是通过期权市场价格反推出来的未来波动预期水平，是期权定价和情绪判断的重要变量。
- 易混概念：
  IV 不是历史波动率，它反映的是市场愿意为未来不确定性支付的价格。
- 在本项目中的作用：
  期权分析、跨资产情绪观察和波动率套利都需要 IV。

### 10.23 Delta-Neutral / Delta 中性

- 中文：Delta 中性
- 英文：Delta-Neutral
- 缩写：无固定缩写
- 详细定义：
  Delta 中性指通过组合不同头寸，使组合对标的价格一阶变动的敏感度接近零，从而更聚焦其他风险因子或收益来源。
- 易混概念：
  Delta 中性不等于完全无风险，因为 Gamma、Vega、跳空和流动性风险仍可能存在。
- 在本项目中的作用：
  期权做市、波动率交易和部分跨品种套利策略需要 Delta 中性框架。

## 11. 情报、情绪与另类数据术语

### 11.1 Market Intelligence / 市场情报

- 中文：市场情报
- 英文：Market Intelligence
- 缩写：Intel
- 详细定义：
  市场情报是对新闻、公告、社交媒体、宏观事件、链上动态、资金流和市场异动等信息源进行实时搜集、结构化和解释后的结果。
- 易混概念：
  市场情报不是原始新闻流，它强调筛选、去噪、分类和可行动性。
- 在本项目中的作用：
  情报模块为情绪评分、事件驱动策略和方向偏置信号提供基础数据。

### 11.2 Sentiment Score / 情绪评分

- 中文：情绪评分
- 英文：Sentiment Score
- 缩写：无固定缩写
- 详细定义：
  情绪评分是对市场文本、舆情、公告或链上行为的情感倾向进行量化后的数值表示，例如利多、中性、利空及其强弱程度。
- 易混概念：
  情绪评分不是涨跌预测本身，它只是方向判断的一类输入。
- 在本项目中的作用：
  情绪评分会被写入情报服务、信号服务和报表系统。

### 11.3 Hotness Score / 热度评分

- 中文：热度评分
- 英文：Hotness Score
- 缩写：无固定缩写
- 详细定义：
  热度评分衡量某个标的、事件或主题在一定时间内受到关注的强度，常结合提及量、传播速度、来源权重和异常增幅计算。
- 易混概念：
  热度高不等于利多，热度只是关注度，不代表方向。
- 在本项目中的作用：
  热度评分可用于选标、事件过滤、异常告警和情绪分析。

### 11.4 Directional Bias / 方向偏置

- 中文：方向偏置、方向倾向
- 英文：Directional Bias
- 缩写：无固定缩写
- 详细定义：
  方向偏置表示系统基于因子、情绪、事件和市场确认后，对未来某一时间窗口内上涨、下跌或震荡的倾向判断。
- 易混概念：
  方向偏置不是确定性预测，它必须绑定时间窗、概率和置信度。
- 在本项目中的作用：
  情报与情绪模块输出的方向偏置可作为策略输入，而不是替代策略。

### 11.5 Confidence / 置信度

- 中文：置信度
- 英文：Confidence
- 缩写：无固定缩写
- 详细定义：
  置信度描述系统对某个判断、分类或信号输出的可信程度，通常结合模型稳定性、数据质量、来源权威性和多源一致性估计。
- 易混概念：
  置信度不等于准确率，它是当前单次输出的自我可信度估计。
- 在本项目中的作用：
  方向偏置、事件识别和情绪评分都需要置信度字段用于风控和策略加权。

### 11.6 Event Taxonomy / 事件分类体系

- 中文：事件分类体系
- 英文：Event Taxonomy
- 缩写：无固定缩写
- 详细定义：
  事件分类体系用于将新闻、公告和市场事件统一映射为结构化类别，例如财报、监管、上币、退市、黑天鹅、链上攻击、宏观数据发布等。
- 易混概念：
  没有稳定的事件分类体系，情报系统很难沉淀统计规律和可复用特征。
- 在本项目中的作用：
  情报服务、报表和事件驱动策略都依赖统一的事件分类。

### 11.7 Catalyst / 触发催化剂

- 中文：催化剂、触发事件
- 英文：Catalyst
- 缩写：无固定缩写
- 详细定义：
  Catalyst 指可能在短时间内显著改变市场预期、流动性或波动率的事件或信息源。
- 易混概念：
  Catalyst 更强调“会推动市场反应的事件”，而不是所有普通资讯。
- 在本项目中的作用：
  事件驱动策略、情绪分析和监控告警应重点识别 Catalyst。

### 11.8 Confirmation / 市场确认

- 中文：市场确认
- 英文：Confirmation
- 缩写：无固定缩写
- 详细定义：
  市场确认是指某个情报或信号是否已经被市场价格、成交量、波动率或订单簿变化所验证。
- 易混概念：
  情报利多但价格无确认，并不意味着应直接交易。确认层是降低噪声的重要步骤。
- 在本项目中的作用：
  情报模块输出信号时，建议将确认状态作为必备字段。

### 11.9 Alternative Data / 另类数据

- 中文：另类数据
- 英文：Alternative Data
- 缩写：Alt Data
- 详细定义：
  另类数据是传统行情和财务数据之外的数据源，例如卫星图像、供应链数据、搜索热度、社交媒体、链上行为和网页抓取数据。
- 易混概念：
  另类数据并不天然有用，其价值取决于质量、时效性、可解释性和合规性。
- 在本项目中的作用：
  增强版平台需要支持另类数据接入、标准化、版本化和特征计算。

## 12. 系统架构、数据库与安全术语

### 12.1 Event-Driven Architecture / 事件驱动架构

- 中文：事件驱动架构
- 英文：Event-Driven Architecture
- 缩写：EDA
- 详细定义：
  事件驱动架构是以事件作为系统状态推进和模块协作的基础模式，模块通过发布与订阅事件解耦协作。
- 易混概念：
  事件驱动不是“全异步”或“没有状态”，它同样需要严格的状态边界和幂等保障。
- 在本项目中的作用：
  回测、模拟盘、实盘和监控都建议围绕统一事件模型设计。

### 12.2 Event Bus / 事件总线

- 中文：事件总线
- 英文：Event Bus
- 缩写：无固定缩写
- 详细定义：
  事件总线是负责分发、传递和持久化事件的中间层，可实现模块间解耦和异步协同。
- 易混概念：
  事件总线不是业务逻辑本身，但它会直接影响时序、延迟和系统稳定性。
- 在本项目中的作用：
  数据接入、策略运行、风控、订单回报、监控和回放会依赖事件总线。

### 12.3 Adapter / 适配器

- 中文：适配器
- 英文：Adapter
- 缩写：无固定缩写
- 详细定义：
  适配器用于把外部交易所、券商、数据商或区块链接口映射成平台内部统一的数据结构和调用语义。
- 易混概念：
  适配器不是简单 SDK 包装，它还承担字段映射、错误转换、时序处理、重试和能力差异屏蔽。
- 在本项目中的作用：
  股票、期货、加密货币的多市场支持都依赖高质量适配器层。

### 12.4 Idempotency / 幂等

- 中文：幂等
- 英文：Idempotency
- 缩写：无固定缩写
- 详细定义：
  幂等表示同一操作重复执行多次后，系统最终结果仍与执行一次一致，不会产生重复副作用。
- 易混概念：
  幂等不等于不重试。恰恰相反，只有实现幂等，系统才更能安全地重试。
- 在本项目中的作用：
  下单、撤单、资金同步、回报处理和任务调度都必须考虑幂等设计。

### 12.5 Snapshot / 状态快照

- 中文：状态快照
- 英文：Snapshot
- 缩写：无固定缩写
- 详细定义：
  状态快照是在某一时刻对系统对象状态的完整记录，用于快速恢复、对账、审计和回放起点构建。
- 易混概念：
  快照不是事件明细，它更像某时刻的全量状态截面。
- 在本项目中的作用：
  订单簿、持仓、账户、组合和策略状态都需要周期性快照。

### 12.6 Replay / 事件回放

- 中文：事件回放
- 英文：Replay
- 缩写：无固定缩写
- 详细定义：
  事件回放是按原始事件顺序重新驱动系统状态，用于复盘、验证和故障排查。
- 易混概念：
  本节的 Replay 偏工程语境，第 6 节的 Replay 偏研究与测试语境，两者本质一致但侧重点不同。
- 在本项目中的作用：
  回放是验收、问题定位、策略复盘和 shadow 对比的关键能力。

### 12.7 Shadow Deployment / 影子部署

- 中文：影子部署
- 英文：Shadow Deployment
- 缩写：Shadow
- 详细定义：
  影子部署是让新版本系统在真实输入下并行运行，但不直接影响真实交易结果，用于比较新旧系统行为差异。
- 易混概念：
  Shadow 不是灰度发布。灰度会影响部分真实用户，Shadow 通常不直接影响真实订单。
- 在本项目中的作用：
  策略升级、执行算法替换和风控规则调整前，建议先进入 Shadow 阶段。

### 12.8 Worker / 工作进程

- 中文：工作进程、任务执行器
- 英文：Worker
- 缩写：无固定缩写
- 详细定义：
  Worker 是负责异步执行具体任务的进程或服务，例如因子计算、回测任务、数据抓取和报表生成。
- 易混概念：
  Worker 不等于微服务。一个微服务可以内部包含多个 Worker。
- 在本项目中的作用：
  因子物化、实验任务、情报抓取和对账任务都需要 Worker 体系。

### 12.9 Scheduler / 调度器

- 中文：调度器
- 英文：Scheduler
- 缩写：无固定缩写
- 详细定义：
  调度器负责按时间、依赖关系、优先级和资源约束触发任务运行。
- 易混概念：
  调度器不是任务本身，它决定任务何时、何地、以什么参数运行。
- 在本项目中的作用：
  数据同步、因子计算、模型重训、报表生成和健康检查都需要调度器。

### 12.10 DDL / 数据定义语言

- 中文：数据定义语言
- 英文：Data Definition Language
- 缩写：DDL
- 详细定义：
  DDL 用于描述数据库对象结构，例如表、索引、约束、分区和视图。
- 易混概念：
  DDL 解决“结构怎么定义”，不负责“数据怎么写入”。
- 在本项目中的作用：
  本项目后续需要将文档级 schema 落为可执行 DDL。

### 12.11 OpenAPI / 接口规范

- 中文：接口规范
- 英文：OpenAPI Specification
- 缩写：OpenAPI
- 详细定义：
  OpenAPI 是描述 REST 接口路径、参数、响应、认证和错误码的标准规范，常用于文档生成和代码生成。
- 易混概念：
  OpenAPI 是契约，不等于实现代码，但良好的契约能显著降低前后端和服务间协作成本。
- 在本项目中的作用：
  增强版控制面接口最终应落为 OpenAPI 契约。

### 12.12 Schema / 模式

- 中文：模式、结构定义
- 英文：Schema
- 缩写：无固定缩写
- 详细定义：
  Schema 可以指数据库模式、消息结构、接口对象结构或配置结构，含义依上下文而定。
- 易混概念：
  讨论 Schema 时必须说明是数据库 Schema、API Schema 还是事件 Schema。
- 在本项目中的作用：
  本项目同时存在数据库 Schema、API Schema 和事件 Schema，需要明确区分。

### 12.13 Primary Key / Foreign Key / Unique Key / 主键、外键、唯一键

- 中文：主键、外键、唯一键
- 英文：Primary Key、Foreign Key、Unique Key
- 缩写：PK、FK、UK
- 详细定义：
  主键用于唯一标识记录，外键用于表达表间引用关系，唯一键用于约束字段或字段组不可重复。
- 易混概念：
  唯一键不一定是主键，一个表可以有多个唯一键但只能有一个主键定义。
- 在本项目中的作用：
  交易、持仓、回报、快照和实验记录都需要清晰的主外键关系。

### 12.14 JSONB / 半结构化字段

- 中文：JSONB 字段
- 英文：JSONB
- 缩写：JSONB
- 详细定义：
  JSONB 是 PostgreSQL 中的二进制 JSON 类型，适合存储结构不完全固定但仍需查询与索引的半结构化数据。
- 易混概念：
  JSONB 灵活但不能替代规范建模；核心交易状态不宜过度依赖 JSONB。
- 在本项目中的作用：
  外部原始回报、扩展元数据、模型解释输出等场景适合使用 JSONB。

### 12.15 Hypertable / 时序超表

- 中文：时序超表
- 英文：Hypertable
- 缩写：无固定缩写
- 详细定义：
  Hypertable 是时序数据库扩展中的概念，用于将大规模时间序列按时间和空间维度自动分片与优化。
- 易混概念：
  它更适合行情、指标和监控数据，不适合复杂事务型订单状态管理。
- 在本项目中的作用：
  行情、因子值和监控指标存储可考虑使用 Hypertable。

### 12.16 RBAC / 基于角色的访问控制

- 中文：基于角色的访问控制
- 英文：Role-Based Access Control
- 缩写：RBAC
- 详细定义：
  RBAC 是通过角色而非直接对个人赋权的权限控制方法，用于统一管理读取、写入、审批和操作权限。
- 易混概念：
  RBAC 解决“谁能做什么”，但不自动解决“是否需要多人审批”。
- 在本项目中的作用：
  生产下单、参数调整、模型上线和风控开关都必须纳入 RBAC 控制。

### 12.17 Approval Flow / 审批流

- 中文：审批流
- 英文：Approval Flow
- 缩写：无固定缩写
- 详细定义：
  审批流是对高风险操作引入多人确认、顺序审查和状态记录的治理机制。
- 易混概念：
  审批流是治理机制，不是权限模型的替代品，两者需要同时存在。
- 在本项目中的作用：
  模型上线、策略启停、风控阈值放宽和生产密钥操作应纳入审批流。

### 12.18 Audit Log / 审计日志

- 中文：审计日志
- 英文：Audit Log
- 缩写：无固定缩写
- 详细定义：
  审计日志用于记录关键对象和关键操作的变更人、变更时间、变更前后值、来源系统和关联审批记录。
- 易混概念：
  应用日志不等于审计日志。审计日志强调不可否认性和可追溯性。
- 在本项目中的作用：
  安全、合规、问题复盘和责任界定都需要完整审计日志。

### 12.19 Runbook / 运维操作手册

- 中文：运维操作手册、运行手册
- 英文：Runbook
- 缩写：无固定缩写
- 详细定义：
  Runbook 是描述异常诊断、人工接管、停机流程、恢复流程、回滚流程和常规运维步骤的操作文档。
- 易混概念：
  Runbook 不是架构设计文档，它强调“出问题时具体怎么做”。
- 在本项目中的作用：
  生产运行、值班响应和故障处理都需要 Runbook。

### 12.20 SLO / SLA / 服务等级目标与协议

- 中文：服务等级目标、服务等级协议
- 英文：Service Level Objective、Service Level Agreement
- 缩写：SLO、SLA
- 详细定义：
  SLO 是内部可量化运行目标，例如接口成功率和延迟；SLA 是对外承诺的服务协议。
- 易混概念：
  很多项目只有监控没有 SLO，导致“看得见问题，但没有明确标准判断是否退化”。
- 在本项目中的作用：
  控制面 API、行情接入、订单链路和监控告警都应设置 SLO。

## 13. 测试、验收与发布术语

### 13.1 Unit Test / 单元测试

- 中文：单元测试
- 英文：Unit Test
- 缩写：UT
- 详细定义：
  单元测试验证单个函数、类或模块在受控输入下的行为是否符合预期，强调快速、稳定和低依赖。
- 易混概念：
  单元测试不是端到端测试，它不负责覆盖完整业务链路。
- 在本项目中的作用：
  因子计算、费用模型、风控规则、订单状态机等都需要充分单元测试。

### 13.2 Integration Test / 集成测试

- 中文：集成测试
- 英文：Integration Test
- 缩写：IT
- 详细定义：
  集成测试用于验证多个模块、服务或外部系统之间的协作是否正确，例如“策略 -> 风控 -> OMS -> 适配器 -> 回报”链路。
- 易混概念：
  通过单元测试不代表集成一定正确，接口契约、时序和异常处理往往只能在集成测试中暴露。
- 在本项目中的作用：
  交易平台的关键链路必须构建系统性集成测试。

### 13.3 Replay Test / 回放测试

- 中文：回放测试
- 英文：Replay Test
- 缩写：无固定缩写
- 详细定义：
  回放测试通过重放历史市场事件、订单回报和系统事件，验证策略和系统在已知历史场景下的行为是否稳定一致。
- 易混概念：
  回放测试更强调事件顺序和状态演化，不只是对收益指标做一次再计算。
- 在本项目中的作用：
  回放测试是执行链路、订单状态机和故障复盘的重要测试手段。

### 13.4 Fault Injection / 故障注入

- 中文：故障注入
- 英文：Fault Injection
- 缩写：无固定缩写
- 详细定义：
  故障注入是主动制造网络抖动、接口超时、重复回报、数据乱序、节点宕机等异常情况，以验证系统韧性。
- 易混概念：
  没有异常测试的“正常测试通过”并不能说明实盘系统可靠。
- 在本项目中的作用：
  行情接入、OMS、风控和监控告警都需要故障注入测试。

### 13.5 UAT / 用户验收测试

- 中文：用户验收测试
- 英文：User Acceptance Testing
- 缩写：UAT
- 详细定义：
  UAT 由业务侧、研究侧或运营侧根据预定义验收标准确认系统是否满足真实使用要求。
- 易混概念：
  UAT 不只是“再跑一遍测试”，它是业务层面对可用性和正确性的最后确认。
- 在本项目中的作用：
  生产前的关键版本必须经过 UAT，尤其是实盘链路和高风险功能。

### 13.6 Acceptance Criteria / 验收标准

- 中文：验收标准
- 英文：Acceptance Criteria
- 缩写：AC
- 详细定义：
  验收标准是判断功能是否可交付的明确条件，通常包括功能正确性、性能阈值、风险边界、日志审计和异常处理要求。
- 易混概念：
  没有量化的验收标准，测试结论就容易变成主观判断。
- 在本项目中的作用：
  每个核心功能都应建立与测试用例相映射的验收标准。

### 13.7 Canary Release / 金丝雀发布

- 中文：金丝雀发布
- 英文：Canary Release
- 缩写：Canary
- 详细定义：
  金丝雀发布是先把新版本投放到少量流量、少量账户或受控环境中观察，再逐步扩大范围的发布方式。
- 易混概念：
  金丝雀发布和影子部署不同，金丝雀会真正影响部分生产流量。
- 在本项目中的作用：
  控制面接口、报表服务和低风险策略服务可先采用金丝雀发布。

### 13.8 Rollback / 回滚

- 中文：回滚
- 英文：Rollback
- 缩写：无固定缩写
- 详细定义：
  回滚是在新版本引发错误或风险时，恢复到上一稳定版本的技术和流程能力。
- 易混概念：
  回滚不仅是代码回退，还可能涉及配置、模型、数据库变更和任务状态恢复。
- 在本项目中的作用：
  所有关键模块上线前都必须有明确回滚方案。

## 14. 易混概念辨析

### 14.1 Factor 与 Feature

- 因子更强调金融意义、解释性和统计优势来源。
- 特征更强调工程表达、模型输入和服务化管理。
- 在本项目里，优秀因子最终通常会被工程化为 Feature。

### 14.2 Signal 与 Target Position 与 Order

- Signal 是“想做什么”的建议。
- Target Position 是“最终想持有多少”的目标状态。
- Order 是“为了达到目标而发出的具体指令”。

### 14.3 OMS 与 EMS

- OMS 负责订单对象和生命周期管理。
- EMS 负责如何拆单、路由和更优成交。
- 没有 OMS 的 EMS 会缺少一致状态源，没有 EMS 的 OMS 会难以优化执行质量。

### 14.4 Backtest 与 Replay 与 Paper Trading

- Backtest 强调历史评估收益和风险。
- Replay 强调重放事件并复现状态变化。
- Paper Trading 强调用实时市场输入检验系统行为但不产生真实资金交割。

### 14.5 Alpha 与 Beta

- Alpha 是超额收益来源。
- Beta 是市场暴露。
- 区分两者是组合归因和策略评价的基本前提。

### 14.6 Liquidity 与 Volume

- 成交量只是流动性的一个表现。
- 流动性还要看点差、深度、成交冲击和订单簿恢复能力。

### 14.7 Mark Price 与 Last Price 与 Index Price

- Last Price 反映最近一笔成交。
- Mark Price 更适合风控和估值。
- Index Price 更适合做参考锚定。

### 14.8 Funding Rate 与 Borrow Rate

- Funding Rate 是永续合约多空之间的资金交换。
- Borrow Rate 是借入资产或资金的融资成本。
- 两者都影响真实收益，但来源不同。

### 14.9 Delta-Neutral 与 Market-Neutral

- Delta 中性强调对标的价格一阶敏感度接近零。
- 市场中性强调整体市场方向暴露接近零。
- 两者相关但不完全等同。

### 14.10 Continuous Contract 与 Main Contract

- 连续合约是研究序列。
- 主力合约是某个时点实际交易最活跃或最被使用的具体合约。
- 回测时需要清楚自己使用的是研究口径还是交易口径。

## 15. 维护原则

- 新增市场支持时，先补术语，再补功能设计。
- 新增数据库表、事件模型、API 对象时，优先检查是否引入了新的核心术语。
- 文档、代码、测试和监控中的关键对象命名，应尽量与本术语表保持一致。
- 如果不同交易所对同一概念定义不同，应在适配器层和专项设计文档中补充分市场说明。

## 16. 结论

这份术语文档的目标不是追求“百科全书式收录所有金融词汇”，而是为本项目建立一套：

- 研究能对齐
- 架构能落地
- 实现能命名一致
- 风控能解释
- 测试能验收
- 运维能复盘

的统一语言基础。

如果后续继续推进实施级文档，最适合与本术语表配套新增的文档是：

- 缩写速查单页版
- 订单与执行状态机术语表
- 风控规则口径词典
- 股票 / 期货 / 加密货币分市场术语补充说明
