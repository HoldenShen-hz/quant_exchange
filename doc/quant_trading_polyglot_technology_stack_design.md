## 1. 文档目标

本文档用于明确 `quant_exchange` 的一个核心技术原则：

不同功能模块可以使用不同的开发语言，只要：

- 模块边界清晰
- 契约统一
- 可测试
- 可部署
- 可审计

量化交易平台并不适合强行把所有模块都塞进一种语言里。研究、回测、因子、NLP、订单执行、风控热路径、控制面 API 这几类模块的优化目标完全不同，更合理的做法是按模块选语言。

## 2. 多语言架构的总体原则

### 2.1 研究效率优先的模块

优先使用：

- Python

适合：

- 因子研究
- 回测
- 策略原型
- NLP / ML
- 数据分析

### 2.2 低延迟和高可靠优先的模块

优先使用：

- Rust

适合：

- OMS / EMS
- 执行网关
- 行情热路径
- 订单簿处理
- 事前风控热路径

### 2.3 平台服务治理优先的模块

优先使用：

- Go

适合：

- 控制面 API
- 配置管理
- 账户同步
- 任务编排
- 运维接口
- 报表 API

### 2.4 前端和控制台

优先使用：

- TypeScript

适合：

- 管理后台
- 报表界面
- 风控控制台

## 3. 推荐的模块语言分工

| 模块 | 推荐语言 | 说明 |
| --- | --- | --- |
| 因子研究与策略原型 | Python | 研究效率最高 |
| 回测引擎 | Python 为主，关键算子可用 Rust 扩展 | 兼顾研究效率和性能 |
| Feature Store 物化作业 | Python + SQL | 数据工程实现方便 |
| NLP / 情绪分析 / 训练 | Python | 模型生态成熟 |
| 实时行情接入 | Rust | 并发和稳定性更强 |
| 订单簿聚合 | Rust | 低延迟、内存安全 |
| OMS / EMS | Rust | 状态机复杂、可靠性要求高 |
| 下单路由与预交易风控 | Rust | 热路径更适合 |
| 控制面 API | Go | 服务治理和部署更稳健 |
| 账户同步 / 对账 | Go | IO 服务较多 |
| 管理后台前端 | TypeScript | 标准 Web 技术栈 |

## 4. 因子计算的语言选择建议

### 4.1 结论

像因子计算这种模块，`Rust` 或 `C / C++` 在纯计算性能上通常确实比 Python 更高。

但真正的工程结论不是“因子计算全部改成 Rust 或 C”，而是：

- 研究和原型阶段：优先 Python
- 高频或超大规模批量因子物化：用 Rust 做性能热点
- 不建议直接大量手写 C，除非已有成熟历史库必须复用

### 4.2 为什么不能简单地说“全部用 Rust / C 更好”

因为因子计算有两类完全不同的工作：

- 研究型因子计算
  - 频繁试验
  - 需要快速改公式
  - 需要和回测、notebook、NLP、ML 联动
- 生产型因子计算
  - 固定公式
  - 需要高吞吐、低延迟、低资源占用
  - 需要持续物化和稳定运行

前者更适合 Python，后者更适合 Rust。

### 4.3 什么时候 Python 足够

以下场景通常 Python 就足够：

- 日频和分钟频因子研究
- 单机回测
- 截面选股 / 选币 / 选品种研究
- Pandas / Polars / NumPy 可向量化的批处理
- 情绪因子和文本因子
- 需要和 notebook、sklearn、PyTorch、XGBoost 联动的流水线

### 4.4 什么时候建议上 Rust

以下场景建议用 Rust 重写热点部分：

- 实时因子流式计算
- 高频订单簿因子
- 秒级或毫秒级滚动聚合
- 超大规模标的池的增量物化
- 同时处理多市场多通道实时特征
- Python 已经明确成为 CPU 或内存瓶颈

### 4.5 C / C++ 和 Rust 怎么选

如果只看性能，Rust、C、C++ 都可以做高性能因子计算。  
但如果从长期维护角度看，建议优先：

- Rust，而不是直接裸用 C

原因：

- Rust 更安全
- 并发更稳
- 内存错误更少
- 更适合长期运行的交易系统

如果团队已经有成熟 C++ 低延迟经验，也可以接受：

- C++ 作为执行层和计算热点语言

但不建议新项目把大量新核心模块建立在裸 C 上。

## 5. 推荐的因子计算分层方案

建议采用三层因子计算架构：

### 5.1 因子定义层

语言：

- Python

职责：

- 定义公式
- 定义窗口
- 定义输入依赖
- 做研究和验证

### 5.2 因子执行层

语言：

- Python 或 Rust

职责：

- 批量物化
- 滚动更新
- 增量计算

建议：

- 普通批量特征先 Python
- 高吞吐热点算子再逐步 Rust 化

### 5.3 因子服务层

语言：

- Go 或 Python

职责：

- 暴露查询接口
- 发布版本
- 触发重算任务

## 6. 多语言服务之间的协作方式

建议统一以下机制：

- OpenAPI
- Protobuf / gRPC
- 统一错误码
- 统一日志字段
- 统一 tracing id
- 统一 domain model

推荐：

- 外部接口：HTTP / JSON
- 内部同步：gRPC / Protobuf
- 异步事件：Redis Streams 或 Kafka

## 7. MVP 落地建议

第一阶段建议只控制在三种主语言：

- Python
- Rust
- Go

推荐分工：

- Python：
  - 因子研究
  - 回测
  - NLP / 情绪分析
- Rust：
  - 实时行情
  - 执行网关
  - OMS / EMS
  - 事前风控热路径
- Go：
  - 控制面 API
  - 账户同步
  - 任务编排
  - 报表和运维接口

## 8. 结论

像因子计算这种模块，`Rust` 或 `C / C++` 的性能通常确实更高，但系统设计上最合理的方式不是“一刀切全部换掉 Python”，而是：

- Python 负责研究、验证、原型和大部分批处理
- Rust 负责性能热点、流式特征和低延迟链路
- Go 负责平台服务和控制面

这比“全 Python”更强，也比“所有东西都用 Rust / C”更现实、更可维护。
