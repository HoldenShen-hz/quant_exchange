## 1. 文档目的

本文档按最新主需求编号体系，对 `quant_exchange` 的各模块做“可开发、可测试、可排期”的详细展开。

本文档回答以下问题：

- 某个需求项在工程上究竟包含哪些子能力
- 需要哪些输入、输出、状态和接口
- 模块边界在哪里，哪些内容不在本模块负责范围内
- 模块完成的最小交付物是什么
- 增强模块在进入排期前还缺哪些前置条件

本文档是主需求文档的展开层，不替代主需求文档。  
若两者冲突，以 [quant_trading_requirements_analysis.md](./quant_trading_requirements_analysis.md) 为准。

## 3. 全局设计约束

### 3.1 统一标识与时间口径

- 全平台内部统一使用 `instrument_id`
- 时间字段必须具有统一比较口径，展示层可按市场或用户时区转换
- 历史数据、回测、模拟盘、实盘、网页查询必须共享统一的标识映射

### 3.2 用户与账户隔离

- 用户状态、学习进度、观察列表、工作台设置、模拟账户、机器人状态、社区内容权限必须按用户隔离
- 多账户能力引入前，不得混淆“用户”和“交易账户”概念

### 3.3 审计与恢复

- 所有关键状态变化必须可审计
- 所有长任务必须具备可恢复状态
- 所有高风险动作必须具备原因、操作者和时间记录

### 3.4 风控优先级

- 风控拥有阻断交易、阻断跟单、阻断机器人操作、阻断工作流动作的最高优先级
- AI、Webhook、Copy Trading、策略市场等增强模块均不得绕过风控

### 3.5 页面与控制面约束

- 页面只负责状态展示和用户交互，不承担业务判定
- 所有可恢复状态都必须由控制面接口提供持久化和查询能力
- 任何页面可见状态都应能通过接口重建

## 4. 核心交易与研究模块详细规格

### 4.1 模块 `8.1 / MD` 市场数据与参考数据

#### 4.1.1 对应需求

`MD-01`、`MD-02`、`MD-03`、`MD-04`、`MD-05`、`MD-06`、`MD-07`、`MD-08`、`MD-09`、`MD-10`

#### 4.1.2 模块职责

负责接入、标准化、存储和分发股票、期货、加密货币相关历史数据、实时数据和参考数据，为策略、回测、模拟、实盘和网页工作台提供统一数据底座。

#### 4.1.3 范围内能力

| 子域 | 详细定义 | 对应需求 |
| --- | --- | --- |
| 数据源接入 | 接入历史数据源、实时行情源、参考数据源 | `MD-01` |
| 历史数据任务 | 全量拉取、增量更新、时间窗口补齐 | `MD-02` |
| 下载任务控制 | 后台下载、断点续传、失败重试、状态查询 | `MD-03` |
| 统一数据模型 | 标的、日线、分钟线、成交、盘口摘要、资金费率、账户快照 | `MD-04` |
| 数据分层 | 原始层、标准层、特征层 | `MD-05` |
| 数据质量 | 去重、乱序、未来时间戳、精度异常、缺失检测 | `MD-06` |
| 查询服务 | 按标的、时间、市场、周期查询 | `MD-07` |
| 全市场快照 | 分钟级汇总股票池 / 市场状态 / 最新价 | `MD-08` |
| 主数据导入 | A 股 / 港股 / 美股统一主数据导入 | `MD-09` |
| 参考数据 | 公司行为、交易日历、主力连续映射 | `MD-10` |

#### 4.1.4 关键数据与状态

- `Instrument`
- `MarketCalendar`
- `CorporateAction`
- `KlineBar`
- `MinuteBar`
- `TradeTick`
- `OrderbookSummary`
- `FundingRate`
- `RealtimeSnapshot`
- `HistoricalDownloadJob`

#### 4.1.5 关键流程

1. 从外部数据源拉取原始数据
2. 写入原始层并执行标准化转换
3. 在标准层进行幂等写入
4. 触发质量校验、异常记录和补齐策略
5. 将可供网页和交易读取的数据分发给上层服务

#### 4.1.6 实现要求

- 标的映射必须独立实现，不得散落在页面或策略代码中
- 下载任务状态必须持久化，支持恢复
- 参考数据不能与行情数据混表存储
- 全市场快照生成必须与前台查询解耦，避免页面阻塞

#### 4.1.7 最小交付物

- 统一标的模型
- 日线与分钟线标准化查询
- 股票主数据导入器
- 历史下载控制器
- 实时快照接口

#### 4.1.8 非目标

- 当前阶段不要求全市场逐笔真行情
- 当前阶段不要求 `L3` 订单簿和专业热力图

### 4.2 模块 `8.2 / ST` 策略研究与开发

#### 4.2.1 对应需求

`ST-01`、`ST-02`、`ST-03`、`ST-04`、`ST-05`、`ST-06`、`ST-07`、`ST-08`

#### 4.2.2 模块职责

提供统一策略模板、参数管理、因子调用、信号输出和运行记录能力，支撑回测、模拟盘和实盘共用策略核心逻辑。

#### 4.2.3 范围内能力

| 子域 | 详细定义 | 对应需求 |
| --- | --- | --- |
| 策略基类 | 统一生命周期和上下文 | `ST-01` |
| 逻辑拆分 | 数据输入、因子、信号、仓位、订单意图 | `ST-02` |
| 参数化配置 | 支持文件 / 控制面配置驱动 | `ST-03` |
| 版本与运行标识 | 版本号、`run_id`、变更记录 | `ST-04` |
| 因子基础库 | 均线、动量、波动率、均值回归等 | `ST-05` |
| 实验记录 | 参数、结果、运行摘要、说明 | `ST-06` |
| 多语言扩展 | Python 主体 + Rust / Go 热点扩展 | `ST-07` |
| 增强入口 | DSL、可视化编辑、自然语言生成 | `ST-08` |

#### 4.2.4 关键数据与状态

- `StrategyDefinition`
- `StrategyParameterSet`
- `StrategyRun`
- `FactorOutput`
- `SignalDecision`
- `TargetPosition`

#### 4.2.5 关键流程

1. 注册策略模板
2. 加载参数配置并生成运行上下文
3. 读取市场数据与账户状态
4. 计算因子与信号
5. 输出目标仓位或订单意图
6. 写入运行记录和关键指标

#### 4.2.6 实现要求

- 回测、模拟盘、实盘必须共享核心策略逻辑
- 参数快照必须在运行开始时冻结
- 公共因子不得复制粘贴到具体策略内部

#### 4.2.7 最小交付物

- `BaseStrategy`
- `StrategyContext`
- 策略注册表
- 参数化配置能力
- 基础因子库

#### 4.2.8 非目标

- 当前阶段不要求完整 Notebook Lab 和分布式实验平台

### 4.3 模块 `8.3 / BT` 回测

#### 4.3.1 对应需求

`BT-01`、`BT-02`、`BT-03`、`BT-04`、`BT-05`、`BT-06`、`BT-07`、`BT-08`

#### 4.3.2 模块职责

在历史数据上重放策略行为，模拟订单、资金、持仓和成本变化，并输出可复现的结果与偏差分析。

#### 4.3.3 范围内能力

| 子域 | 详细定义 | 对应需求 |
| --- | --- | --- |
| 事件驱动 | 以 `bar/tick/event` 驱动策略执行 | `BT-01` |
| 粒度支持 | 日线、分钟级，预留 Tick 扩展 | `BT-02` |
| 成本约束 | 手续费、滑点、成交延迟、保证金 | `BT-03` |
| 订单仿真 | 市价、限价、撤单、部分成交 | `BT-04` |
| 资金仿真 | 现金、冻结资金、可用余额、持仓 | `BT-05` |
| 批量实验 | 参数对比、滚动窗口、样本内外 | `BT-06` |
| 结果留档 | 结果导出、图表、指标存档 | `BT-07` |
| 偏差审计 | 前视偏差、未来函数、时间对齐 | `BT-08` |

#### 4.3.4 关键数据与状态

- `BacktestRun`
- `BacktestOrder`
- `BacktestFill`
- `BacktestPosition`
- `BacktestEquityCurve`
- `BacktestMetrics`
- `BiasAuditResult`

#### 4.3.5 关键流程

1. 加载策略定义和参数
2. 加载历史数据与成本模型
3. 逐事件执行策略
4. 进行订单撮合与账户更新
5. 生成绩效指标和资金曲线
6. 执行偏差检查并留档

#### 4.3.6 实现要求

- 同一数据和参数必须可复现
- 回测结果、参数和版本必须绑定
- 批量实验结果应可汇总比较

#### 4.3.7 最小交付物

- 事件驱动回测器
- 成本模型
- 订单仿真器
- 绩效指标输出
- 偏差检查器

#### 4.3.8 非目标

- 当前阶段不要求复杂订单簿级回测
- 当前阶段不要求分布式回测集群

`PP-01`、`PP-02`、`PP-03`、`PP-04`、`PP-05`、`PP-06`、`PP-07`

#### 4.4.2 模块职责

以实时行情驱动仿真账户，验证策略、执行、风控和页面交互链路。

#### 4.4.3 范围内能力

| --- | --- | --- |

| 行情驱动成交 | 实时行情触发订单仿真 | `PP-02` |
| 订单状态机 | 创建、排队、部分成交、拒单、撤单 | `PP-03` |
| 数据持久化 | 订单、成交、持仓、权益 | `PP-04` |
| 页面交互 | 人工买入、卖出、撤单、重置 | `PP-05` |
| 偏差分析 | 与回测结果对比 | `PP-06` |
| 目标偏差 | 当前模拟仓位 vs 目标仓位 | `PP-07` |

#### 4.4.4 关键数据与状态

- `PaperAccount`
- `PaperOrder`
- `PaperFill`
- `PaperPosition`
- `PaperSnapshot`
- `PaperDriftSummary`

#### 4.4.5 关键流程

1. 获取用户的模拟账户
2. 接收策略或人工下单请求
3. 经风控校验后进入仿真撮合
4. 更新订单、成交、持仓和权益

#### 4.4.7 最小交付物

- 纸面账户服务
- 仿真成交逻辑
- 订单 / 成交 / 持仓查询接口
- 工作台模拟交易页

### 4.5 模块 `8.5 / EX` 实盘执行

#### 4.5.1 对应需求

`EX-01`、`EX-02`、`EX-03`、`EX-04`、`EX-05`、`EX-06`、`EX-07`、`EX-08`

#### 4.5.2 模块职责

负责把策略或人工产生的订单意图转换为真实可执行订单，并管理其状态、重试和异常恢复。

#### 4.5.3 范围内能力

| 子域 | 详细定义 | 对应需求 |
| --- | --- | --- |
| 通道适配 | 交易所 / 券商 / 网关适配 | `EX-01` |
| 基础订单类型 | 市价、限价、撤单 | `EX-02` |
| 状态同步 | 系统内订单状态与外部状态对齐 | `EX-03` |
| 幂等与补偿 | 重试、幂等、超时补偿、对账 | `EX-04` |
| 风控联动 | 执行前统一校验 | `EX-05` |
| 权限控制 | 账户 / 策略 / 标的三级约束 | `EX-06` |
| 异常处理 | 断连、限频、拒单、回报延迟 | `EX-07` |
| 高级执行 | OMS / EMS / 智能路由扩展 | `EX-08` |

#### 4.5.4 关键数据与状态

- `ExecutionRequest`
- `Order`
- `OrderUpdate`
- `Fill`
- `ExecutionAdapterState`
- `ExecutionCompensationTask`

#### 4.5.5 关键流程

1. 接收订单意图
2. 调用权限与风控
3. 通过适配器提交订单
4. 维护内部状态机
5. 补偿未确认订单与遗漏回报
6. 写审计与异常日志

#### 4.5.6 实现要求

- 同一业务请求不得重复下单
- 适配器与 OMS 必须解耦
- 状态同步和人工操作必须统一收敛到同一订单状态机

#### 4.5.7 最小交付物

- 统一执行请求模型
- 订单状态机
- 至少一个交易通道适配器
- 补偿与对账入口

### 4.6 模块 `8.6 / RK` 风险控制

#### 4.6.1 对应需求

`RK-01`、`RK-02`、`RK-03`、`RK-04`、`RK-05`、`RK-06`、`RK-07`

#### 4.6.2 模块职责

在账户、策略、标的和订单四个层面阻断高风险行为，并在异常时触发保护动作。

#### 4.6.3 范围内能力

| 子域 | 详细定义 | 对应需求 |
| --- | --- | --- |
| 账户级规则 | 杠杆、回撤、单日亏损、保证金率 | `RK-01` |
| 策略级规则 | 仓位上限、日损上限、连续亏损熔断 | `RK-02` |
| 标的级规则 | 单标的敞口、流动性、波动率 | `RK-03` |
| 订单级规则 | 数量、金额、价格偏离、异常滑点 | `RK-04` |
| 全局保护 | 停机、减仓、平仓、升级告警 | `RK-05` |
| 黑天鹅保护 | 数据冻结、接口异常、网络异常 | `RK-06` |
| 风控留痕 | 原因码、人工确认、审批轨迹 | `RK-07` |

#### 4.6.4 关键数据与状态

- `RiskRule`
- `RiskDecision`
- `RiskEvent`
- `KillSwitchState`
- `EmergencyAction`

#### 4.6.5 关键流程

1. 接收订单 / 策略动作 / 工作流动作
2. 读取当前账户、持仓、行情与配置
3. 执行分层规则判断
4. 返回放行、裁剪或阻断结果
5. 若超阈值，触发告警和保护动作

#### 4.6.6 实现要求

- 风控规则必须版本化
- 原因码必须稳定，不允许页面自行拼接文案代替
- AI、机器人、跟单、Webhook 统一进入同一风控入口

#### 4.6.7 最小交付物

- 风控规则引擎
- 风控原因码字典
- 全局停机开关
- 风险事件日志

### 4.8 模块 `8.8 / MO` 监控与告警

#### 4.8.1 对应需求

`MO-01`、`MO-02`、`MO-03`、`MO-04`、`MO-05`、`MO-06`

## 6. 需求到交付物的收敛规则

### 6.1 任何模块进入开发前必须明确

- 对应需求编号
- 该模块是否属于 `MVP`
- 该模块的状态持久化边界
- 该模块的接口边界
- 该模块的错误码与异常路径
- 该模块的测试入口

### 6.2 任何增强模块进入排期前必须补齐

- 独立 API 设计
- 独立数据模型与表结构
- 安全与权限边界
- 测试方案与验收标准
- 与现有 `MVP` 链路的隔离策略

### 6.3 完成定义

模块只有满足以下条件，才可判定为“规格完成，可进入编码”：

- 本文档中模块职责和范围已冻结
- 对应主需求编号无歧义
- 输入、输出、状态、流程已明确
- 至少能指向后续一份架构 / 数据 / API / 测试文档

## 8. 结论

本次重构后的详细规格文档，已经和主需求文档、竞品分析文档对齐，不再沿用旧的“模块很多但编号体系不统一”的结构。

现在可以把整套文档理解为三层：

- 主需求文档：定义“做什么”
- 本详细规格文档：定义“怎么拆、怎么交付”
- 测试文档：定义“怎么验证”

后续若继续扩展功能，应优先在主需求文档中增加需求编号，再在本文档中补充对应详细规格，避免再次出现文档结构失配。
