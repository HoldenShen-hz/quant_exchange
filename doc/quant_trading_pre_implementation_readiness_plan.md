## 1. 文档目标

本文档用于回答一个关键问题：

在正式进入代码实现之前，这个量化交易平台项目还需要完成哪些工作，才能降低返工风险、控制实现偏差，并让研发、量化、风控、测试、运维可以按统一标准协作。

自主需求文档 `v2.0` 重构后，本文档默认以新的需求编号体系作为实施准备基线：

- 核心交易与研究模块：`MD / ST / BT / PP / EX / RK / PF / MO / RP / SE / IN / SW / CR / LH / BOT`
- 增强模块：`AI / SOC / COPY / MKT / DSL / VIS / CHART / MOB / HOOK / COMP / ACCT / COLLAB / SCREEN / OPT / FX / TAX`

编码前的冻结项、审批项、状态机和测试计划，均应能映射到以上编号体系。

这份文档不是重复已有的需求、架构、数据库和 API 设计，而是补上“从设计走向实现”之间的最后一层准备工作，重点解决以下问题：

- 哪些事情是编码前必须确认的
- 哪些文档还需要继续细化
- 哪些外部依赖需要先选型和验证
- 哪些规则必须先定，不能边写代码边猜
- 进入开发前，项目应满足什么样的 Go / No-Go 门槛

这些文档已经回答了“系统做什么”“为什么这样设计”“大体怎么拆模块”。

但是在真正编码之前，仍然存在一层必须补齐的实施级准备工作。原因很简单：

- 需求正确，不等于实现口径已经统一
- 架构清晰，不等于状态机和错误处理已经定稿
- schema 完整，不等于 DDL、迁移策略和保留策略已经可执行
- API 文档完整，不等于鉴权、错误码、幂等、异步任务协议已经冻结
- 功能齐全，不等于运营、合规、风控和审批链路已经可落地

因此，编码前必须再完成一轮“实施闭环”。

## 4. 详细实施前工作定义

### 4.1 MVP 边界冻结

#### 为什么必须先做

如果在编码前不冻结边界，开发阶段最容易出现的情况是：

- 产品想做“全平台”
- 研发只能做“最小闭环”
- 风控要求“必须可审计”
- 量化研究希望“一开始就支持多策略、多交易所、多市场”

这会导致范围不断膨胀，最后每个模块都只完成一半。

#### 必须明确的内容

- 第一阶段目标市场是否同时覆盖加密货币市场、标准化期货市场和股票市场
- 第一阶段目标品种是否覆盖加密货币现货、永续、交割合约，期货主力或具体合约，以及 A 股 / 港股 / 美股现金股票
- 第一阶段目标通道是否包含交易所直连、期货柜台 / 经纪商网关，以及股票券商通道
- 第一阶段是否只支持自营账户，不支持多租户 SaaS
- 第一阶段策略范围是否限定为 CTA / 横截面择时 / 情绪增强信号
- 第一阶段是否暂不做期权实盘、做市实盘、DEX 实盘
- 第一阶段情报源是否只接新闻、公告、社交媒体，不接付费另类数据
- 第一阶段是否只支持单币种结算或单主账户体系

#### 应输出的文档或结论

- MVP 范围清单
- 非目标清单
- 第一阶段交易所清单
- 第一阶段策略类型清单
- 第一阶段不支持能力清单

#### 完成标准

- 每个团队对“第一阶段必须做”和“第一阶段明确不做”口径一致
- 排期中的任务都能映射到 MVP 范围
- 没有需要“以后再说”的核心边界问题

### 4.2 外部依赖与数据源选型

#### 为什么必须先做

量化平台不是纯内部系统，它高度依赖外部接口。交易所 API、新闻源、社交平台、消息队列、时序数据库、对象存储等一旦选错，后面重构成本极高。

#### 必须确认的决策

- 行情源使用交易所原生 WebSocket 还是聚合行情服务
- 期货柜台或经纪商网关采用何种接入协议
- 股票市场采用哪类券商或执行通道
- 股票市场数据源是否包含公司行为、复权因子和停复牌信息
- 历史数据回补使用交易所 REST 还是独立历史数据供应商
- 新闻源是否采购商业源，还是先接 RSS / 官网公告
- 社交媒体数据是否通过官方 API 获取，还是先做公开可用源
- 消息系统使用 Kafka、Redis Streams 还是更轻量方案
- 时序数据是否落 PostgreSQL + Timescale，还是冷热分层
- 对象存储使用本地文件、MinIO 还是云对象存储
- 模型训练与特征计算使用单机任务队列还是分布式调度

#### 需要形成的交付物

- 供应商和技术选型对比表
- 成本评估
- 限频和速率限制说明
- 服务稳定性和 SLA 假设
- 交易所接口差异表
- 数据许可与使用约束表

#### 完成标准

- 每个外部依赖都有明确的接入方案和替代方案
- 已识别关键限流、封禁、断连、字段不一致、时间延迟风险
- 可以开始写统一 adapter，而不是边接边改模型

### 4.3 统一领域模型与数据字典

#### 为什么必须先做

编码阶段最常见的隐患，不是“不会写”，而是不同模块对同一概念理解不同。例如：

- `symbol` 到底使用 `BTCUSDT`、`BTC-USDT` 还是内部标准写法
- `position_side` 到底是 long / short，还是 net
- `pnl` 到底是 realized / unrealized / total
- 时间戳到底是事件时间、采集时间、入库时间还是交易所时间

如果不先统一，后面回测、实盘、报表一定会对不齐。

#### 必须统一的对象

- 市场标识、交易所标识、账户标识、子账户标识
- 交易对和合约标识
- 合约面值、合约乘数、到期时间、结算资产、交易日历
- 股票代码、市场区域、计价货币、结算周期、公司行为、卖空能力
- 订单、成交、持仓、资产、资金流水对象
- 信号、目标仓位、订单意图、执行回报对象
- 因子、特征、标签、模型版本、实验对象
- 情报文档、情绪分数、方向偏置、置信度对象
- 风控事件、审批事件、人工干预事件对象

#### 需要形成的交付物

- 统一数据字典
- 枚举值字典
- 字段命名规范
- 时间语义规范
- 精度和舍入规范
- 资产估值和汇率换算口径

#### 完成标准

- 数据库字段名、API 字段名、消息事件字段名之间口径一致
- 关键枚举不依赖个人习惯
- 订单、持仓、PnL、净值指标的定义可以被测试和风控直接引用

### 4.4 风控规则手册

#### 为什么必须先做

风控不是一个“后置模块”，而是编码阶段所有模块的前提约束。如果没有先把规则写清楚，策略、执行、账户、报表都会反复重写。

#### 必须先定的规则

- 单策略最大仓位
- 单标的最大仓位
- 单账户最大总敞口
- 杠杆上限
- 保证金预警阈值
- 日内亏损阈值
- 周度和月度回撤阈值
- 单次下单金额上限
- 连续失败次数熔断阈值
- 下单频率限制
- 集中度限制
- 黑名单资产或市场限制
- 情绪信号参与决策时的最大权重
- 高波动时是否自动降杠杆或禁开新仓

#### 必须明确的执行口径

- 风控检查发生在信号后、下单前、下单中还是成交后
- 风控阻断后是否允许人工 override
- 谁可以 override
- override 是否必须审批
- kill switch 由谁触发
- stop trading 与 flatten positions 是否是两个不同动作

#### 需要形成的交付物

- 风控规则手册
- 规则优先级说明
- 风控错误码和拦截原因字典
- kill switch 操作说明
- 人工干预操作记录模板

#### 完成标准

- 风控可以被编码成规则引擎或前置校验链
- 风控拒单原因能完整落库和展示
- 测试团队可以据此设计拦截和放行用例

### 4.5 权限矩阵与审批流

#### 为什么必须先做

量化平台里最危险的不是 bug 本身，而是“谁可以在什么情况下做什么”没有边界。尤其是以下操作：

- 策略上线
- 参数修改
- API key 更换
- 风控阈值调整
- 手动下单
- 手动平仓
- kill switch
- 恢复交易

#### 必须先明确的角色

- 超级管理员
- 平台管理员
- 量化研究员
- 策略开发者
- 交易员
- 风控专员
- 审计人员
- 只读观察者

#### 必须明确的审批对象

- 新策略上线
- 已上线策略参数调整
- 风控规则变更
- 交易所账户绑定和解绑
- 杠杆上限提升
- 人工接管实盘

#### 需要形成的交付物

- RBAC 权限矩阵
- 操作审批流程图
- 高风险操作双人复核规范
- 审计日志必填字段清单
- 权限最小化原则说明

#### 完成标准

- 每个高风险动作都能映射到角色、权限、审批、审计
- 平台不存在“任何管理员都能直接做所有事”的灰区
- API 和前端操作都能复用同一权限口径

### 4.6 订单与执行状态机设计

#### 为什么必须先做

订单执行是最容易因状态理解不一致而产生资金风险的模块。尤其是不同交易所对以下行为定义可能不同：

- `new`
- `accepted`
- `partially_filled`
- `filled`
- `canceled`
- `expired`
- `rejected`
- `cancel_pending`
- `replace_pending`

#### 必须明确的内容

- OMS 母订单状态机
- EMS 子订单状态机
- 下单失败后的重试规则
- 幂等键设计
- cancel / replace 语义
- 部分成交后的目标仓位修正逻辑
- 断线重连后的订单对账逻辑
- 订单补单、漏单、重复回报的处理逻辑
- 交易所 ack 缺失时的超时策略

#### 需要形成的交付物

- OMS 状态机文档
- EMS 状态机文档
- 订单事件字典
- 重试与补偿规则
- 订单对账规则
- 交易所特殊行为差异说明

#### 完成标准

- 任一订单都可以被完整追溯
- 状态机可以支持自动化测试
- 实盘时不会因为重复消息或网络抖动导致重复下单

### 4.7 数据治理与数据质量规则

#### 为什么必须先做

量化平台中，数据问题造成的损失往往大于代码语法错误。没有统一的数据治理规范，后续回测和实盘会持续出现“信号看起来有效，但其实建立在脏数据上”的问题。

#### 必须明确的内容

- 原始数据、标准化数据、特征数据、回测数据的分层
- 7 x 24 市场和分时段交易市场的时间口径统一
- 历史回补规则
- 缺口检测规则
- 重复数据去重规则
- 异常值处理规则
- 数据订正和回写规则
- 数据版本管理规则
- 数据保留周期和归档策略
- 时区和夏令时口径
- 时间同步策略
- 主力合约映射和展期规则
- 股票公司行为复权和除权口径

#### 对情报与情绪模块特别需要补充的规则

- 信息源可信度评分
- 重复新闻和转载聚合规则
- 情绪标签的人工校验机制
- 多语言文本处理规则
- 事件时间与发布时间的区分
- 模型输出与人工标签的评估口径

#### 需要形成的交付物

- 数据治理文档
- 数据质量 SLA 和质量指标
- 缺口与异常检测规则
- 数据版本和回补流程
- 情报源可信度和去重规则

#### 完成标准

- 回测、研究、实盘引用的数据口径一致
- 数据异常可被自动发现和告警
- 关键数据修复后能够追踪版本影响范围

### 4.8 OpenAPI 契约与接口治理

#### 为什么必须先做

只有“接口列表”还不够。编码前必须把接口真正冻结成机器可校验的契约，否则前后端、服务间联调阶段会大量返工。

#### 必须明确的内容

- 鉴权方式
- access token 生命周期
- 内部服务鉴权方式
- 请求幂等键规范
- 分页规范
- 过滤、排序、时间范围查询规范
- 同步接口和异步任务接口的边界
- 导出类任务的回调或轮询协议
- 错误码体系
- 限流策略
- API 版本管理规则

#### 需要形成的交付物

- OpenAPI 草案
- 请求响应模型
- 错误码字典
- 鉴权说明
- 异步任务协议说明
- API 版本策略说明

#### 完成标准

- 后端可据此生成或校验接口
- 测试可据此编写接口用例
- 前端与自动化调用方不需要靠口头解释理解接口

### 4.9 可执行 DDL、迁移与数据初始化策略

#### 为什么必须先做

schema 文档解决的是设计问题，DDL 和迁移策略解决的是工程落地问题。没有可执行 DDL，项目无法真正进入工程实施。

#### 必须明确的内容

- PostgreSQL 版本和扩展版本
- schema 命名规范
- 主键策略
- 分区或 hypertable 策略
- 索引策略
- 唯一约束和幂等约束
- 审计字段标准
- 软删除与硬删除边界
- 初始 seed data 策略
- migration 工具和命名规范

#### 需要形成的交付物

- 可执行 PostgreSQL DDL
- 初始迁移脚本
- schema 变更规范
- 初始化数据脚本
- 表保留周期和归档策略

#### 完成标准

- 可在本地和测试环境一键初始化数据库
- schema 迁移可回放、可审计、可回滚
- 关键表索引与约束已经过基本审阅

### 4.10 测试方案与验收用例

#### 为什么必须先做

量化系统很容易“能跑但不可信”。测试计划必须在编码前确定，否则开发阶段会默认只验证 happy path。

#### 必须覆盖的测试层次

- 单元测试
- 服务集成测试
- 交易所适配器测试
- 历史数据回放测试
- 回测一致性测试
- 模拟盘联调测试

### 4.11 安全基线与密钥管理

#### 为什么必须先做

交易系统的安全不是“上线前补一下”的事情。API key、资金操作权限、管理员权限、日志脱敏，如果一开始不定规范，后面几乎必然返工。

#### 必须明确的内容

- API key、secret、passphrase 的存储方式
- 环境变量、密钥管理系统或本地加密方案
- 生产、测试、开发环境隔离
- 敏感日志脱敏规则
- 管理后台登录与二次验证要求
- IP 白名单策略
- 主账户与子账户隔离策略
- 审计日志不可篡改要求

#### 需要形成的交付物

- 安全基线文档
- 密钥管理规范
- 日志脱敏规范
- 环境隔离规范
- 高风险操作安全要求

#### 完成标准

- 开发环境也不会到处散落真实凭证
- 高风险操作具备最小权限和审计留痕
- 日志、报错、监控不会泄露敏感字段

### 4.12 监控、告警与运行手册

#### 为什么建议在开发前定义

即使编码可以先开始，如果没有提前想好怎么观测系统，很多关键埋点在开发后期会被遗漏。

#### 必须明确的监控对象

- 行情延迟
- 数据缺口
- WebSocket 断连率
- 下单成功率
- 撤单成功率
- 订单确认超时
- 部分成交滞留
- 对账差异
- 风控拦截次数
- 策略异常停止
- 情报抓取失败率
- 情绪模型服务延迟
- 任务队列积压
- 数据库慢查询

#### 需要形成的交付物

- 指标清单
- 告警矩阵
- 故障处置 Runbook
- 值班响应流程
- 恢复流程和回滚流程

#### 完成标准

- 核心链路都有观测点
- 告警不会泛滥，也不会遗漏高风险事件
- 运维和研发能按同一手册处置事故

### 4.13 工程脚手架与开发规范

#### 为什么必须尽早确定

如果项目一开始没有统一工程规范，代码写得越多，后期整治成本越高。

#### 必须明确的内容

- 本地开发启动方式
- 任务执行框架
- 日志规范
- lint / format / type check 规则
- 测试命令和覆盖率目标
- 分支和合并策略
- CI 流水线最小门槛

#### 需要形成的交付物

- 工程约定文档
- 项目脚手架结构说明
- CI / CD 基线
- 配置分层规范
- 本地开发环境初始化说明

#### 完成标准

- 新同事可以在短时间内拉起项目
- 多人同时开发不会大量踩命名和结构冲突
- 提交到主干前有自动质量门禁

### 4.14 模型验证与情绪信号治理

#### 为什么需要单独拿出来

本项目已经引入实时情报和市场情绪分析模块，这一能力很有价值，但也最容易被误用。必须明确：

- 情绪信号是辅助因子，不是绝对预测器
- 模型分数是概率型、偏置型输出，不是“必涨必跌”的结论
- 任何情绪模型都必须持续评估漂移、误报和迟滞

#### 必须明确的内容

- 标签定义方法
- 预测时间窗口
- 评估指标
- 置信度解释方式
- 人工复核规则
- 模型回滚策略
- 多源信息冲突时的处理方式
- 情绪分数在策略中的最大影响权重

#### 需要形成的交付物

- 模型验证标准
- 情绪信号使用规范
- 漂移检测规则
- 误报与回顾分析模板
- 模型降级和停用流程

#### 完成标准

- 任何模型上线前都有验证结论
- 信号效果恶化时有自动告警和降级策略
- 业务团队不会把模型输出误读为确定性结论

### 4.15 发布、回滚与实盘准入策略

#### 为什么必须提前明确

量化平台不能采取“开发完就上线”的发布方式。必须有从研究到回测、从模拟盘到小资金实盘、再到放量运行的渐进式准入路径。

- 放量阈值
- 回滚触发条件
- 自动停机与人工接管边界
- 实盘参数变更流程

#### 需要形成的交付物

- 发布流程文档
- 回滚清单
- 实盘准入标准
- 小流量和灰度方案
- 事故后复盘模板

#### 完成标准

- 每个策略都必须经过标准化准入流程
- 不是由个人经验决定是否可以放量
- 出现异常时能快速降级和回滚

## 6. 推荐的编码前实施顺序

建议不要把所有事情都一次性拉满，而是按下面顺序推进。

### 阶段 0：决策冻结

目标是先把不会随着代码推进而自然解决的问题定下来。

- 冻结 MVP 范围
- 确认交易所和数据源
- 确认领域模型和术语口径
- 确认风控规则
- 确认权限和审批要求

### 阶段 1：契约冻结

目标是把未来所有协作边界变成文档和契约。

- 输出 OpenAPI 草案
- 输出可执行 DDL
- 输出 adapter 接入规范
- 输出订单状态机
- 输出测试和验收案例

### 阶段 2：工程就绪

目标是让项目真正可以开始持续编码和集成。

- 完成仓库脚手架
- 建立 CI 基线
- 建立本地开发环境
- 建立 observability 基线
- 建立发布和回滚流程

## 7. Go / No-Go 检查清单

在进入主干开发前，至少应满足以下检查项：

| 检查项 | 是否必须 | 说明 |
| --- | --- | --- |
| MVP 范围是否冻结 | 必须 | 否则排期和实现边界持续漂移 |
| 目标交易所是否确定 | 必须 | 否则 adapter 和数据模型无法稳定 |
| 目标数据源是否确定 | 必须 | 否则数据层会反复返工 |
| 领域模型是否统一 | 必须 | 否则回测、实盘、报表难以对齐 |
| 风控规则是否形成书面规则 | 必须 | 否则下单链路无法闭环 |
| 权限矩阵是否定义 | 必须 | 否则高风险操作不可控 |
| 审批流是否定义 | 必须 | 否则实盘操作不可审计 |
| OMS / EMS 状态机是否冻结 | 必须 | 否则订单链路风险过高 |
| DDL 是否可执行 | 必须 | 否则数据库无法真正落地 |
| API 契约是否冻结 | 必须 | 否则联调阶段返工很大 |
| 测试策略是否制定 | 必须 | 否则只能验证 happy path |
| 密钥管理是否定义 | 必须 | 否则存在严重安全风险 |
| 监控和告警是否定义核心指标 | 建议 | 否则后期补埋点成本高 |
| 发布和回滚策略是否定义 | 建议 | 否则实盘准入不可控 |
| 工程脚手架是否搭好 | 建议 | 否则多人协作成本高 |

如果上表中任意一个“必须”项尚未明确，原则上不建议直接进入大规模编码。

## 8. 如果跳过这些工作的主要风险

如果在编码前忽略上述准备工作，项目最可能出现以下问题：

- 回测结果和实盘结果严重偏离
- 不同模块对订单、持仓、PnL 口径不一致
- 数据质量问题在上线后才暴露
- 风控逻辑分散在多个服务中，无法审计
- 高风险操作没有审批和留痕
- 交易所适配器反复重构
- API 接口在联调阶段持续修改
- 数据库字段和约束在开发中频繁变更
- 情绪模型被误用为强预测器，造成过度交易
- 出现事故后没有统一恢复路径

## 9. 结论

在当前阶段，项目已经完成了“平台蓝图”层面的主体设计，但在正式编码前，还需要补齐“实施级闭环”。

最关键的原则不是尽快写代码，而是先把那些一旦写错就会导致系统性返工或资金风险的问题定下来。对于量化交易平台来说，编码前最值得投入时间的，不是多写一个模块，而是先冻结以下几类东西：

- 边界
- 契约
- 规则
- 权限
- 状态机
- 验收口径

只有这些前置工作完成，后续代码实现才能真正做到：

- 开发可并行
- 测试可验证
- 风控可追责
- 实盘可放量
- 事故可回滚

因此，本项目下一阶段的最优动作不是盲目开工，而是先把 P0 级实施前文档补齐，再进入代码骨架和核心链路实现。
